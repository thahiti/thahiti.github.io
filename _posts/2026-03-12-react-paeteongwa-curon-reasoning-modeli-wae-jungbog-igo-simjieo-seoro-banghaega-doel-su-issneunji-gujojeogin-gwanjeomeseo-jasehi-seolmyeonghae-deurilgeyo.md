ReAct 패턴과 추론(Reasoning) 모델이 왜 '중복'이고, 심지어 서로 방해가 될 수 있는지 구조적인 관점에서 자세히 설명해 드릴게요.
이해를 돕기 위해 비유하자면, **ReAct는 '속으로 생각할 줄 모르는 사람에게 억지로 소리 내어 생각하게 만드는 훈련법'**이고, **추론 모델은 '이미 머릿속에서 깊고 복잡하게 생각하는 능력을 갖춘 전문가'**입니다. 이 둘을 결합하면 여러 가지 비효율과 충돌이 발생합니다.
1. 생각 과정의 중복 (External CoT vs Internal CoT)
 * ReAct의 방식 (외부적 사고): ReAct 프롬프트는 모델에게 Thought(생각) -> Action(행동)의 형식을 강제합니다. 모델은 텍스트를 한 글자씩 출력하면서(소리 내어 말하면서) 다음 행동을 결정하는 논리를 스스로 만들어갑니다.
 * 추론 모델의 방식 (내부적 사고): o1이나 DeepSeek-R1 같은 추론 모델은 답변을 텍스트로 출력하기 전에, 이미 내부적으로 '숨겨진 생각의 사슬(Hidden Chain of Thought)'을 거칩니다. 수많은 시뮬레이션과 논리적 검증을 내부에서 끝낸 뒤에야 최종 텍스트를 내뱉습니다.
 * 중복의 결과: 추론 모델에게 ReAct 포맷을 요구하면, 모델은 '어떻게 ReAct의 Thought: 포맷에 맞춰서 글을 쓸지'를 또 내부적으로 깊게 고민하게 됩니다. 즉, 생각을 하기 위해 생각하는 이중 구조가 되어버려 컴퓨팅 자원과 시간만 낭비됩니다.
2. 형식 준수(Formatting)의 충돌
ReAct 패턴이 에이전트로서 작동하려면 외부 도구(API, 검색 등)와 연결되어야 합니다. 이를 위해서는 모델의 출력이 컴퓨터가 파싱(Parsing)하기 쉬운 완벽한 형태를 갖춰야 합니다.
> 일반 모델 (GPT-4o, Claude 3.5 Sonnet 등): 지시사항을 잘 따릅니다. "반드시 Action: search("날씨") 형식으로만 대답해"라고 하면 군더더기 없이 정확히 그 형식만 출력하여 도구가 즉시 실행될 수 있게 합니다.
>
> 추론 모델: 형식 준수에 매우 취약합니다. 이들은 '정해진 틀에 맞추는 것'보다 '문제의 논리적 해결'에 최적화되도록 강화학습(RL)을 받았습니다. 따라서 단답형이나 정해진 JSON 포맷만 내놓으라고 해도, 혼자 논리적 전개를 펼치며 수십 줄의 설명을 덧붙이는 경향이 있습니다. 이로 인해 에이전트의 코드(파서)가 모델의 출력을 인식하지 못하고 에러를 내는 경우가 빈번하게 발생합니다.
>
3. 속도와 비용(Latency & Cost)의 기하급수적 증가
ReAct 루프는 기본적으로 다중 턴(Multi-turn) 구조입니다.
생각 -> 행동 -> 관찰(결과 받음) -> 다시 생각 -> 행동...
하나의 문제를 풀기 위해 LLM을 5~10번씩 연속해서 호출하는 일이 흔합니다.
 * 일반 모델은 한 번 호출할 때 응답이 빠르기 때문에 이 루프가 비교적 원활하게 돌아갑니다.
 * 하지만 추론 모델은 한 번 대답할 때마다 내부적으로 긴 사고 과정을 거치므로 속도가 느리고 API 비용도 비쌉니다. 이 느린 과정을 ReAct 루프 안에서 매 턴마다 반복하게 되면, 간단한 작업 하나를 끝내는 데 몇 분씩 걸리게 되어 실용성이 크게 떨어집니다.
💡 요약하자면
ReAct는 과거에 일반 LLM들이 복잡한 논리적 추론을 잘하지 못할 때, 그 한계를 극복하기 위해 만들어진 프롬프트 엔지니어링 기법입니다. 반면 최근의 추론 모델들은 그 논리적 추론 능력을 아예 모델 구조와 학습 단계에 내재화해버렸습니다. 따라서 이미 해결된 문제를 위한 기법(ReAct)을 최신 추론 모델에 다시 억지로 끼워 맞출 필요가 없는 것입니다.
추론 모델의 등장으로 인해 에이전트 설계 방식이나 프롬프트를 작성하는 패러다임이 많이 바뀌고 있습니다. 혹시 추론 모델(o1, DeepSeek-R1 등)의 성능을 극대화하기 위해 가장 권장되는 새로운 프롬프트 작성법이나 구조에 대해 더 알아보고 싶으신가요?
