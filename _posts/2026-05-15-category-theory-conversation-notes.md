# 범주론 학습 대화 전체 기록

---

## 대상과 사상의 프로그래밍 대응

범주론의 **대상(Object)**과 **사상(Morphism)**은 프로그래밍에서 다음에 대응됨:

**대상 = 타입(Type)**
`Int`, `String`, `List[Int]`, `Option[String]` 같은 타입 자체가 대상이다. 값(value)이 아니라 타입이라는 점이 핵심.

**사상 = 함수(Function)**
타입 A에서 타입 B로 가는 함수 `f: A → B`가 사상이다. 예를 들어 `parseInt: String → Int`는 `String` 대상에서 `Int` 대상으로 가는 화살표(사상).

이걸 범주론의 법칙에 대입하면:

- **합성**: `f: A → B`와 `g: B → C`가 있으면 `g ∘ f: A → C`가 반드시 존재 → 함수 합성 (`compose`, pipe)
- **항등 사상**: 모든 타입 A에 `id: A → A` (입력을 그대로 반환하는 identity 함수)가 존재
- **결합 법칙**: `h ∘ (g ∘ f) = (h ∘ g) ∘ f` → 함수 합성 순서를 어떻게 묶든 결과 동일

이 구조를 프로그래밍 언어론에서는 **Hask** (Haskell의 타입과 함수로 이루어진 범주)라고 부름. Python이나 TypeScript도 개념적으로 동일한 범주를 형성한다고 볼 수 있는데, 다만 부수 효과나 bottom type(`⊥`, 무한루프/예외) 때문에 수학적으로 엄밀한 범주는 아님.

---

## 항등 사상은 동일한 타입을 반환하면 되는가?

아니, 타입만 같으면 되는 게 아니라 **입력값을 그대로 반환**해야 함.

```python
# 항등 사상 (identity) ✅
def id(x: A) -> A:
    return x

# 같은 타입을 반환하지만 항등 사상이 아닌 예 ❌
def not_id(x: int) -> int:
    return x + 1
```

핵심 조건은 **합성의 항등원** 역할을 해야 한다는 것:

- `f ∘ id = f` (어떤 함수 뒤에 id를 붙여도 원래 함수와 동일)
- `id ∘ f = f` (id 뒤에 어떤 함수를 붙여도 원래 함수와 동일)

`x + 1`은 이 조건을 깨뜨림. 모노이드에서 덧셈의 항등원이 `0`인 것처럼 (어떤 수에 0을 더해도 그 수 자체), 함수 합성의 항등원은 "아무것도 안 하는 함수"여야 함.

---

## 항등 사상의 정의는 값 기반이 아니다

범주론 자체는 값을 안 봄. 항등 사상의 정의도 값 기반이 아니라 **합성 관계**로만 정의됨:

> 모든 사상 `f: A → B`에 대해 `f ∘ id_A = f`이고 `id_B ∘ f = f`를 만족하는 사상

즉 "값을 그대로 반환한다"는 건 범주론의 정의가 아니라, 프로그래밍 범주(Hask)에서 이 합성 조건을 만족하는 함수가 **결과적으로** `return x`일 수밖에 없다는 것.

범주론 관점에서는 `id_A`가 내부적으로 뭘 하는지 관심 없고, 다른 화살표와 합성했을 때 그 화살표를 변화시키지 않는다는 **외부적 관계**만 봄.

---

## Void와 absurd 함수

`Void`는 **값이 하나도 없는 타입**임. (빈 집합 ∅에 대응)

`absurd :: Void → a`는 Void를 받아서 아무 타입 `a`를 반환하는 함수인데, 핵심은:

- Void 타입의 값은 존재하지 않음
- 따라서 이 함수를 **호출할 방법이 없음**
- 호출할 수 없으니 반환 타입이 뭐든 상관없음 → 아무 타입이나 약속해도 모순이 안 생김

논리학(Curry-Howard 대응)으로 보면:

- 타입 = 명제, 값 = 증명
- `Void` = 거짓(False) — 증명(값)이 존재하지 않으므로
- `Void → a` = "거짓이면 무엇이든 참" — 논리학에서 항상 성립하는 명제

이게 라틴어 "ex falso sequitur quodlibet" (폭발 원리). "거짓으로부터는 아무 결론이나 도출할 수 있다"는 뜻. 전제가 애초에 성립 불가능하면 결론이 뭐든 명제 전체는 참이 됨.

실용적으로는 패턴 매칭에서 "이 분기는 절대 도달할 수 없음"을 타입 수준에서 증명할 때 쓰임.

---

## 거짓이면 왜 참인가 — 조건문의 진리표

논리학에서 `P → Q` (조건문)의 진리표를 보면:

| P | Q | P → Q |
|---|---|-------|
| T | T | **T** |
| T | F | **F** |
| F | T | **T** |
| F | F | **T** |

P가 거짓이면 Q가 뭐든 `P → Q`는 **항상 참**.

직관적으로 이해하려면: "→"는 "약속"임.

> "비가 오면 우산을 가져갈게"

이 약속이 **깨지는** 경우는 딱 하나 — 비가 왔는데 우산을 안 가져간 경우(T → F). 비가 안 오면? 우산을 가져가든 말든 약속을 **어긴 게 아님**. 그래서 참.

`Void → a`도 마찬가지. "Void 값을 주면 a를 돌려줄게"라는 약속인데, Void 값을 줄 수 있는 상황 자체가 없으니 이 약속은 절대 어길 수 없음 → 항상 참.

---

## absurd라는 이름의 유래

"absurd"는 영어로 "불합리한, 있을 수 없는"이라는 뜻인데, 논리학의 증명 기법 **귀류법(Reductio ad absurdum)**에서 온 이름.

귀류법은: 어떤 가정이 모순(absurdity)으로 이끌리면, 그 가정은 거짓이라고 결론짓는 방식.

`absurd` 함수는 이걸 뒤집어서 — "Void 값이 존재한다"는 것 자체가 이미 모순(absurd)인 상황이므로, 거기서부터는 아무 결론이나 끌어낼 수 있다는 의미. "불합리한 전제에서 출발했으니 뭐든 가능하다"는 뉘앙스.

---

## 반환 타입에 대한 다형성

`absurd :: Void → a`에서 `a`가 어떤 타입이든 될 수 있다는 뜻.

보통 함수는 반환 타입이 고정됨:

```haskell
length :: String → Int        -- 항상 Int 반환
show   :: Int → String         -- 항상 String 반환
```

그런데 `absurd`는:

```haskell
absurd :: Void → Int       -- 도 되고
absurd :: Void → String    -- 도 되고
absurd :: Void → Bool      -- 도 됨
```

호출자가 원하는 타입 아무거나 반환 타입으로 쓸 수 있음. 이게 "반환 타입에 대한 다형성(parametric polymorphism)".

가능한 이유는 앞서 말한 것과 같음 — 어차피 호출될 일이 없으니 실제로 뭘 반환할지 구현할 필요가 없고, 그래서 아무 타입이나 약속해도 모순이 안 생김.

## 순서 관계를 카테고리로 표현하기 — Thin Category

"≤" 관계를 범주론으로 보는 예시.

대상은 숫자들(1, 2, 3, ...), 사상은 "a ≤ b"라는 관계. 이게 범주가 되려면 세 조건을 확인해야 함:

1. **항등 사상**: `a ≤ a` 성립? → 모든 수는 자기 자신 이하이므로 ✅
2. **합성**: `a ≤ b`이고 `b ≤ c`이면 `a ≤ c`? → 추이성(transitivity)으로 ✅
3. **결합법칙**: 자동으로 성립 ✅

이렇게 ≤ 관계가 있는 집합을 **원순서(Preorder)**라고 하고, 범주의 조건을 만족하니 범주로 볼 수 있음.

여기서 조건을 추가하면 더 강한 구조가 됨:

- **부분순서(Partial Order)**: 원순서 + 반대칭성. `a ≤ b`이고 `b ≤ a`이면 `a = b`. 예를 들어 집합의 포함 관계(⊆)가 이것. {1,2}와 {2,3}처럼 서로 비교 불가능한 원소 쌍이 있을 수 있음.
- **전순서(Total Order)**: 부분순서 + 아무 두 원소든 반드시 비교 가능. 자연수의 ≤가 대표적. 1과 2 중 누가 큰지 항상 말할 수 있음.

요점은 범주론이 숫자와 함수만의 이야기가 아니라, "≤" 같은 순서 관계도 대상과 사상으로 표현할 수 있다는 것. 범주론의 추상화 능력을 보여주는 예시.

---

## 원순서(Preorder)란

집합 위에 관계 ≤가 있을 때, 딱 두 가지만 만족하면 원순서(Preorder)임:

1. **반사성(Reflexivity)**: 모든 a에 대해 `a ≤ a`
2. **추이성(Transitivity)**: `a ≤ b`이고 `b ≤ c`이면 `a ≤ c`

그게 전부. "pre-order"라서 "순서 이전의 것", 즉 순서의 가장 약한 형태라는 뜻.

부분순서(Partial Order)와의 차이는 **반대칭성**이 없다는 것. 즉 `a ≤ b`이고 `b ≤ a`인데 `a ≠ b`인 상황이 허용됨.

현실 예시: "A 도시에서 B 도시로 가는 길이 있다"는 관계를 생각해봐. A→B 길도 있고 B→A 길도 있지만 A와 B는 다른 도시 — 이게 원순서에서는 가능함. 부분순서였으면 "서로 갈 수 있으면 같은 도시여야 한다"는 제약이 붙음.

---

## 반사성, 추이성 등의 개념은 어디서 나왔는가

집합론(Set Theory)과 그 위에 세워진 **순서론(Order Theory)**에서 나온 개념들임. 더 넓게는 **관계 이론(Theory of Relations)**이라는 분야.

수학에서 "관계(Relation)"란 집합의 원소들 사이에 성립하는 조건을 말함. 예를 들어 자연수 집합에서 ≤, <, =, "나눈다" 같은 것들이 전부 관계임. 19세기 말~20세기 초에 칸토어(Cantor), 데데킨트(Dedekind) 등이 집합론을 정립하면서 관계의 성질들을 분류하기 시작했고, 그때 반사성·추이성·대칭성·반대칭성 같은 이름이 붙음.

이 성질들을 조합하면 다양한 관계 구조가 나오는데, 반사성+추이성이면 원순서, 거기에 반대칭성을 추가하면 부분순서, 비교가능성까지 추가하면 전순서가 되는 식임. 이건 범주론보다 훨씬 오래된 기초 수학이고, 범주론이 1940년대에 등장하면서 "이런 순서 구조도 범주로 볼 수 있네"라고 재해석한 것.

## 세 가지 순서 완결 설명

### 공통 기반

세 가지 순서는 전부 "집합 위에 정의된 관계"임. 관계란 집합의 원소들 사이에 "이것과 저것이 이런 사이다"를 정해주는 규칙이고, 이 관계가 어떤 성질들을 만족하느냐에 따라 이름이 달라짐. 사용되는 성질은 네 가지:

- **반사성**: 모든 a에 대해 `a ≤ a` (자기 자신과 항상 관계 성립)
- **추이성**: `a ≤ b`이고 `b ≤ c`이면 `a ≤ c` (관계가 전이됨)
- **반대칭성**: `a ≤ b`이고 `b ≤ a`이면 `a = b` (서로 관계가 성립하면 같은 것)
- **완전성**: 임의의 두 원소 a, b에 대해 `a ≤ b`이거나 `b ≤ a` 중 반드시 하나는 성립 (모든 쌍이 비교 가능)

### 원순서 (Preorder) = 반사성 + 추이성

가장 느슨한 형태. 반대칭성이 없기 때문에 `a ≤ b`이고 `b ≤ a`인데 `a ≠ b`인 상황이 허용됨.

**예시 1 — 도시 간 도로**: "A에서 B로 가는 도로가 존재한다"는 관계. 모든 도시는 자기 자신에게 갈 수 있고(반사성), A→B 도로와 B→C 도로가 있으면 A→C로 갈 수 있다고 치면(추이성) 원순서. 서울→부산 도로도 있고 부산→서울 도로도 있지만 서울 ≠ 부산(반대칭성 불만족).

**예시 2 — 동급 직급 관계**: 회사에서 "A가 B에게 업무 지시를 할 수 있다"는 관계. 팀장 김씨와 팀장 이씨가 서로 지시를 내릴 수 있는 동급이지만, 김씨 ≠ 이씨. 반대칭성이 성립하지 않으므로 원순서.

### 부분순서 (Partial Order) = 반사성 + 추이성 + 반대칭성

원순서에 반대칭성이 추가됨. "서로 관계가 성립하면 같은 것"이라는 제약이 붙기 때문에 원소 간에 구별이 명확해짐. 하지만 완전성이 없기 때문에 **비교 불가능한 쌍이 존재할 수 있음**.

**예시 1 — 집합의 포함관계(⊆)**: {1}, {2}, {1,2}, {1,2,3}이라는 집합들이 있을 때, {1} ⊆ {1,2} ⊆ {1,2,3}은 비교 가능하지만, {1}과 {2}는 어느 쪽도 다른 쪽을 포함하지 않으므로 비교 불가능. A ⊆ B이고 B ⊆ A이면 A = B이므로 반대칭성도 만족. 따라서 부분순서.

**예시 2 — 양의 정수의 나눗셈 관계**: `a | b` ("a가 b를 나눈다"). 2 | 6이고 6 | 24이면 2 | 24(추이성). 모든 수는 자기 자신을 나눔(반사성). a | b이고 b | a이면 a = b(양의 정수에서 반대칭성 성립). 하지만 3과 5는? 3이 5를 나누지도, 5가 3을 나누지도 않음 → 비교 불가능. 따라서 부분순서.

**예시 3 — 프로그래밍의 타입 상속**: `Animal ← Dog ← Poodle`, `Animal ← Cat`이라는 상속 구조에서 "A가 B의 상위타입이다"는 관계. Poodle → Dog → Animal로 추이적이고, 반사적이고, 반대칭적. 하지만 Dog과 Cat은 상속 관계가 없으므로 비교 불가능. 부분순서.

### 전순서 (Total Order) = 반사성 + 추이성 + 반대칭성 + 완전성

부분순서의 모든 조건에 완전성이 추가됨. 아무 두 원소를 골라도 반드시 비교가 가능하기 때문에 모든 원소를 한 줄로 일렬 나열할 수 있음.

**예시 1 — 자연수의 ≤**: 아무 두 자연수를 골라도 반드시 크기 비교가 됨. 3과 7 → 3 ≤ 7. 100과 2 → 2 ≤ 100. 비교 불가능한 쌍이 존재하지 않음.

**예시 2 — 시간의 선후관계**: "사건 A가 사건 B보다 먼저 일어났거나 동시에 일어났다." 단일 시간축 위의 사건들은 반드시 비교 가능함.

**예시 3 — 사전식 순서(Lexicographic order)**: 문자열을 사전에서 찾을 때의 순서. "apple"과 "banana" 중 뭐가 앞인지 항상 결정 가능. 아무 두 문자열이든 반드시 순서를 매길 수 있으므로 전순서.

### 한 눈에 보기

**원순서**: 반사성 + 추이성. 가장 느슨함. "서로 관계가 있는데 다른 것"이 허용됨. 예: 도시 간 도로, 동급 직급 간 관계.

**부분순서**: 원순서 + 반대칭성. "서로 관계가 있으면 같은 것"이라는 제약 추가. 하지만 비교 불가능한 쌍이 존재할 수 있음. 예: 집합의 ⊆, 나눗셈 관계, 타입 상속.

**전순서**: 부분순서 + 완전성. 모든 쌍이 비교 가능. 원소들을 한 줄로 나열 가능. 예: 자연수의 ≤, 시간 순서, 사전식 순서.

각 단계에서 조건이 하나씩 추가되며 구조가 더 엄격해지는 포함 관계임: 전순서 ⊂ 부분순서 ⊂ 원순서.

---

## "구조"란 무엇이며, ≤에서만 정의되는가?

### "구조"란

수학에서 "구조"는 집합 + 그 위에 정의된 규칙(연산이나 관계)을 합쳐서 부르는 말임. 집합 자체는 그냥 원소들의 모임일 뿐인데, 거기에 "원소들끼리 이런 규칙이 성립한다"는 걸 얹으면 구조가 됨. 예를 들어 자연수 집합 {1, 2, 3, ...}은 그냥 집합이지만, 거기에 ≤ 관계를 얹으면 "전순서 구조를 가진 집합"이 되는 것.

### ≤에서만 정의되는 게 아님

원순서, 부분순서, 전순서는 **≤ 기호가 아니라 성질로 정의**되는 것임. 어떤 관계든 반사성과 추이성을 만족하면 원순서고, 거기에 반대칭성까지 만족하면 부분순서임.

예를 들어 "나눈다(divisibility)" 관계 `a | b` ("a가 b를 나눈다")를 보면:

- 반사성: `a | a` → 모든 수는 자기 자신을 나눔 ✅
- 추이성: `a | b`이고 `b | c`이면 `a | c` ✅
- 반대칭성: `a | b`이고 `b | a`이면 `a = b` (양의 정수에서) ✅
- 완전성: 3과 5는? 3이 5를 나누지도, 5가 3을 나누지도 않음 ❌

그래서 양의 정수 위의 나눗셈 관계는 **부분순서**임. ≤ 기호를 전혀 쓰지 않았지만, 같은 성질들을 만족하니까 같은 구조인 것.

또 다른 예로, 프로그래밍에서 클래스의 상속 관계도 부분순서임. `Animal ← Dog ← Poodle` 관계에서 Poodle은 Dog의 하위타입이고 Dog는 Animal의 하위타입이니 추이적으로 Poodle은 Animal의 하위타입(추이성). 모든 타입은 자기 자신의 하위타입(반사성). 그런데 `Dog`과 `Cat`은 서로 상속 관계가 없으니 비교 불가능 → 부분순서.

### 그래서 "순서"라는 이름이 붙은 이유

이 구조들에 "순서(order)"라는 이름이 붙은 건, 이 성질들을 만족하는 관계가 원소들 사이에 "누가 앞이고 누가 뒤다"라는 방향성을 만들어내기 때문임. ≤는 그 중 가장 익숙한 사례일 뿐이고, 핵심은 기호가 아니라 반사성·추이성·반대칭성·완전성이라는 **성질의 조합**이 어떤 종류의 "순서 매기기"가 가능한지를 결정한다는 것.

---

## 부분순서에서 비교 불가능한 쌍은 반대칭성 때문인가?

직접적인 상관은 없음.

**반대칭성**은 "비교 불가능한 쌍이 존재한다"와 관계없음. 반대칭성은 `a ≤ b`이고 `b ≤ a`이면 `a = b`라는 조건일 뿐임. 즉 "서로 포함하면 같은 거다"라는 얘기지, 비교 불가능성과는 별개.

비교 불가능한 쌍이 존재하는 이유는 **완전성(Totality) 조건이 없기** 때문임. 전순서는 "아무 두 원소든 반드시 비교 가능"이라는 완전성을 요구하는데, 부분순서는 그걸 요구하지 않음. 그래서 {1}과 {2}처럼 비교 안 되는 쌍이 허용되는 것.

정리하면 부분순서가 "부분"인 이유는 반대칭성 때문이 아니라, 완전성이 빠져있기 때문임. 반대칭성은 부분순서를 원순서와 구분짓는 조건이고, 완전성의 부재가 부분순서를 전순서와 구분짓는 조건임.

---

## 직관적 예시를 통한 세 가지 순서 이해

**원순서** — SNS 팔로우 관계

A가 B를 팔로우하고, B가 A를 팔로우할 수 있음. 근데 A ≠ B. 서로 맞팔한다고 같은 사람이 되는 건 아님. 자기 자신은 당연히 팔로우한 거나 마찬가지(반사성), A→B 팔로우하고 B→C 팔로우하면 A→C도 팔로우할 수 있다고 치면(추이성) 원순서. 가장 느슨함.

**부분순서** — 회사 조직도

사원 → 대리 → 과장 → 부장 이런 수직 라인이 있음. 위에서 아래로 지시 가능하고, 서로 지시할 수 있으면 같은 직급(반대칭성). 그런데 마케팅팀 과장과 개발팀 과장은? 서로 지시할 수 없음. 누가 위고 아래인지 말할 수 없는 쌍이 존재. 비교 가능한 것도 있고 불가능한 것도 있음.

**전순서** — 줄 서기

놀이공원에서 한 줄로 서 있는 사람들. 아무 두 사람을 골라도 "누가 앞이고 누가 뒤"인지 반드시 말할 수 있음. 비교 불가능한 쌍이 없음. 모든 사람이 일렬로 나열되어 있으니까.

---

## 수를 이용한 예제

**원순서** — "차이가 3 이하"라는 관계

숫자 1, 2, 3, 4, 5에서 "a와 b의 차이가 3 이하면 a ≤ b로 간주"라고 정의하면: 2→4 성립, 4→2도 성립. 근데 2 ≠ 4. 서로 관계가 있는데 다른 원소임. 반대칭성 불만족이니 원순서.

**부분순서** — 약수 관계

{1, 2, 3, 6, 12}에서 "a가 b를 나눈다"는 관계:

- 1 → 2 → 6 → 12 ✅ 비교 가능
- 1 → 3 → 6 → 12 ✅ 비교 가능
- 2와 3? 2가 3을 나누지도, 3이 2를 나누지도 않음 → **비교 불가능**

일부는 비교 가능하고 일부는 안 됨. 부분순서.

**전순서** — 그냥 ≤

{1, 2, 3, 6, 12}에서 "a ≤ b" (크기 비교):

- 1 ≤ 2 ≤ 3 ≤ 6 ≤ 12
- 2와 3? → 2 ≤ 3 ✅
- 아무 두 수를 골라도 반드시 비교 가능

같은 집합 {1, 2, 3, 6, 12}인데 어떤 **관계**를 얹느냐에 따라 부분순서가 되기도 하고 전순서가 되기도 함. 구조를 결정하는 건 원소가 아니라 관계임.

---

## 관계는 사상이 아닌데 왜 범주론에 가져오는가

바로 그게 포인트임. 관계는 원래 사상이 아닌데, **관계를 사상으로 볼 수 있다**는 게 범주론의 추상화 능력을 보여주는 것.

원순서 집합을 범주로 해석하면:

- 대상 = 집합의 각 원소 (예: 1, 2, 3)
- 사상 = `a ≤ b`라는 관계가 성립하면 a에서 b로 가는 화살표가 "존재한다"고 간주

이때 사상의 내용은 중요하지 않음. "a에서 b로 가는 화살표가 있느냐 없느냐"만 따짐. `a ≤ b`가 참이면 화살표 있음, 거짓이면 없음. 그래서 두 대상 사이에 사상이 최대 하나만 존재하는 특수한 범주가 됨(이걸 thin category라고 부름).

이렇게 재해석하면 원순서의 성질이 범주의 공리와 정확히 대응됨:

- 반사성 `a ≤ a` → 항등 사상 존재
- 추이성 `a ≤ b, b ≤ c → a ≤ c` → 사상의 합성 존재

이 문서에서 순서론을 가져온 이유는, "범주론이 함수와 타입만 다루는 게 아니라 순서 관계, 논리, 집합 등 전혀 다른 수학 분야의 구조도 자기 프레임워크 안에 통합할 수 있다"는 걸 보여주기 위함임. 범주론을 "수학의 수학"이라고 부르는 이유가 바로 이것 — 서로 다른 분야에서 반복되는 패턴을 "대상과 사상"이라는 하나의 언어로 통일해서 다룸.

---

## Thin Category에서 사상이 최대 하나인 이유

원순서에서 관계 `a ≤ b`는 **성립하거나 안 하거나** 둘 중 하나임. "a ≤ b가 두 가지 다른 방식으로 성립한다"는 건 없음.

이걸 범주로 번역하면:

- `a ≤ b`가 참 → a에서 b로 가는 화살표가 **하나** 존재
- `a ≤ b`가 거짓 → a에서 b로 가는 화살표가 **없음** (공집합)

그래서 두 대상 사이에 사상이 **최대 1개**. 0개 아니면 1개.

반면 일반적인 범주에서는 같은 두 대상 사이에 사상이 여러 개 있을 수 있음. 예를 들어 집합 {1, 2}에서 {a, b}로 가는 함수는 (1→a, 2→a), (1→a, 2→b), (1→b, 2→a), (1→b, 2→b)로 4개. 같은 두 대상 사이에 사상이 여러 개이므로 thin이 아님.

관계를 범주로 만들면 **항상 thin**임. Thin하지 않은 범주를 얻으려면, 관계가 아니라 함수처럼 같은 출발점과 도착점 사이에 여러 가지 "방법"이 존재하는 사상을 써야 함.

---

## 프로그래밍에서 타입을 대상으로 택하는 이유

범주론은 "내부 구조를 안 보고 관계만 본다"는 철학이기 때문.

값 수준에서 범주를 만들면 1→1, 1→2, 2→3, ... 이렇게 개별 값 사이의 매핑을 전부 따져야 함. 이건 결국 집합론처럼 내부 원소를 하나하나 들여다보는 거랑 같아짐. 범주론이 추상화를 통해 얻으려는 이점이 사라짐.

타입 수준에서 보면 `Int → String`이라는 화살표 하나가 "정수를 문자열로 바꾸는 모든 가능한 방법"을 대표함. 개별 값이 어떻게 매핑되는지는 사상 내부의 구현 디테일이고, 범주론은 그 화살표가 **존재하는지**, **합성이 가능한지**만 관심 가짐.

실용적인 이유도 있음. 타입 수준에서 범주를 구성해야 함자(Functor), 모나드 같은 개념이 자연스럽게 나옴. `List`는 타입을 받아서 타입을 반환하는 것이지 (`Int → List[Int]`), 값을 받아서 값을 반환하는 게 아님. `map`, `flatMap` 같은 연산도 "이 타입의 함수를 저 타입의 함수로 변환한다"는 타입 수준의 이야기임.

요약하면: 값은 대상의 "내부 원소"이고, 범주론은 내부를 안 보는 학문이라서 타입(대상) 사이의 함수(사상)만 다루는 것.

---

## 순서론에서는 값을 직접 사용하는데?

맞음. 범주론에서 "대상"이 뭐냐는 **범주마다 다름**.

- 순서론 범주: 대상 = 개별 값(1, 2, 3), 사상 = ≤ 관계
- 프로그래밍 범주(Hask): 대상 = 타입(Int, String), 사상 = 함수

범주론은 "대상은 반드시 타입이어야 한다" 같은 규칙이 없음. 대상과 사상이 뭐든 상관없이, 항등 사상과 합성이 존재하고 결합법칙을 만족하면 범주임.

프로그래밍에서 굳이 타입을 대상으로 택한 이유는, 그렇게 해야 함자·모나드 같은 유용한 도구가 자연스럽게 나오기 때문임. 값을 대상으로 삼는 범주를 만들 수도 있지만, 거기서는 `List`, `Maybe`, `flatMap` 같은 프로그래밍에 쓸모 있는 구조가 나오지 않음.

즉 "어떤 것을 대상으로 삼을까"는 **어떤 문제를 풀고 싶은가**에 따른 선택인 것. 순서를 연구하고 싶으면 값을 대상으로 삼고, 타입 시스템과 부수 효과를 다루고 싶으면 타입을 대상으로 삼는 것.

---

## 프로그래머에게 순서 구조가 중요한 이유 — 정렬 알고리즘

정렬 알고리즘이 동작하려면 "아무 두 원소를 비교할 수 있어야" 함. 그게 전순서.

**전순서 → 일반 정렬 알고리즘**

숫자 [3, 1, 4, 2]를 정렬할 때 퀵소트든 버블소트든 항상 두 원소를 비교함: "3과 1 중 뭐가 큼?" → 항상 답할 수 있음. 모든 쌍이 비교 가능하니까 한 줄로 쭉 나열 가능: [1, 2, 3, 4].

**부분순서 → 비교 불가능한 쌍이 존재**

약수 관계에서 2와 3은 비교 불가능. 이런 데이터를 퀵소트에 넣으면 "2와 3 중 뭐가 앞이야?"라는 질문에 답할 수 없어서 알고리즘이 제대로 작동 안 함. 한 줄로 나열이 불가능.

대신 **위상 정렬(Topological Sort)**을 써야 함. 위상 정렬은 "비교 가능한 쌍의 순서만 지키면서" 나열하는 것. 예를 들어 빌드 시스템에서 모듈 A가 B에 의존하고, A가 C에도 의존할 때, B와 C는 서로 의존 관계가 없음(비교 불가능). 위상 정렬은 "B, C를 A보다 먼저"만 보장하고, B와 C의 순서는 아무렇게나 둠: [B, C, A]도 되고 [C, B, A]도 됨.

프로그래머가 데이터의 순서 구조를 파악하는 게 중요한 이유: 전순서면 일반 정렬, 부분순서면 위상 정렬, 잘못 판단하면 알고리즘 선택을 틀리게 됨.

---

## 사상에 전달인자가 없는 이유

사상에 "전달인자"라는 개념은 없음. 사상은 함수가 아니라 **화살표**임.

범주론에서 사상 `f: A → B`는 "A에서 B로 가는 화살표가 존재한다"는 것이지, "A라는 값을 넣으면 B라는 값이 나온다"가 아님. 출발 대상과 도착 대상이 있을 뿐, 뭔가를 "넣는" 개념이 아님.

항등 사상도 마찬가지. 항등 사상 `id_A: A → A`는 "A에서 A로 가는 화살표"일 뿐이고, 이게 특별한 이유는 다른 화살표와 합성했을 때 그 화살표를 변화시키지 않는다는 **합성 관계** 때문임.

항등원은 모노이드 같은 대수적 구조에서 나오는 개념이고, 거기는 "연산의 인자"가 있음. 예를 들어 덧셈에서 `a + 0 = a`이면 0이 항등원. 여기서 `+`는 두 값을 받는 연산임.

범주론은 이보다 한 단계 더 추상적임. 연산의 인자나 값의 내부를 안 보고, 오직 "화살표들이 어떻게 연결되느냐"만 봄. 프로그래밍에서 사상을 함수로 **해석**할 때 비로소 인자 개념이 생기는 거지, 범주론 자체에는 인자가 없음.

---

## 범주론의 다양한 해석

범주론의 대상과 사상을 무엇으로 채우느냐에 따라 다양한 해석이 나옴.

**Set (집합의 범주)**
대상 = 집합, 사상 = 집합 간의 함수. 가장 고전적인 해석. {1,2,3}에서 {a,b}로 가는 함수들이 사상.

**Ord (순서의 범주)**
대상 = 개별 원소, 사상 = 순서 관계(≤). 화살표가 최대 1개인 thin category.

**Hask (프로그래밍의 범주)**
대상 = 타입, 사상 = 함수. 여기서 함자·모나드가 나옴. `Int → String` 같은 함수가 사상.

**논리학 (Curry-Howard 대응)**
대상 = 명제, 사상 = 증명. `A → B`는 "A가 참이면 B도 참"이라는 증명. `Void → a`(absurd)가 여기서 나온 것.

**위상수학 (Top)**
대상 = 위상 공간, 사상 = 연속 함수. "구조를 깨뜨리지 않는 변환"이 사상.

**선형대수 (Vect)**
대상 = 벡터 공간, 사상 = 선형 변환. 행렬 곱이 사상의 합성에 대응.

**관계형 DB**
대상 = 테이블(스키마), 사상 = 테이블 간 변환(쿼리). 자연 조인이 합성에 대응한다는 연구가 있음.

**물리학 (Cobordism)**
대상 = 공간의 경계, 사상 = 시공간의 진화. 양자역학의 상태 변환을 범주론으로 기술.

핵심은 전부 동일한 규칙(항등 사상 존재, 합성 가능, 결합법칙)을 만족하기 때문에, 한 범주에서 발견한 정리가 다른 범주에서도 그대로 적용된다는 것. 이게 범주론을 "수학의 수학"이라고 부르는 이유임.

---

## 모노이드(Monoid)의 도입

### 범주론에서 모노이드는 대상이 하나뿐인 범주

대상이 하나(★)밖에 없으니까 모든 사상이 `★ → ★`임. 이 상황에서:

- 사상들의 **합성**이 이항 연산 역할을 함 (사상 f와 g를 합성하면 새로운 사상 g∘f)
- **항등 사상** `id_★`이 항등원 역할을 함 (어떤 사상과 합성해도 그 사상 자체)
- **결합법칙**은 범주의 공리에서 이미 보장됨

이게 대수학에서 말하는 모노이드(집합 + 결합적 이항 연산 + 항등원)와 정확히 같은 구조임.

예를 들어 자연수 덧셈 모노이드를 범주로 보면:

- 대상: ★ 하나
- 사상들: 0, 1, 2, 3, ... (각 자연수가 화살표)
- 합성: 3 ∘ 2 = 5 (덧셈)
- 항등 사상: 0 (아무 수에 더해도 그 수 자체)

왜 모나드 얘기에서 모노이드가 나오냐면, 모나드의 정의가 "자기 함자 범주에서의 모노이드"이기 때문. 자기 함자들을 사상으로 보고, 함자의 합성을 이항 연산으로 보면, 거기서 모노이드 구조를 만족하는 놈이 모나드인 것. `unit`이 항등원, `join`이 이항 연산에 대응함.

---

## 모노이드의 기원 — 추상대수학

**추상대수학(Abstract Algebra)**에서 나온 개념임.

19세기 말~20세기 초에 군(Group) 이론이 발전하면서, 군의 조건 중 "역원의 존재"를 빼면 뭐가 남는지를 연구하다가 모노이드가 정의됨.

- **군(Group)**: 집합 + 이항 연산 + 결합법칙 + 항등원 + **역원**
- **모노이드(Monoid)**: 집합 + 이항 연산 + 결합법칙 + 항등원 (역원 없음)

군에서 조건을 하나 뺀 더 약한 구조인 셈.

범주론(1940년대)이 등장한 후에, "모노이드를 대상이 하나인 범주로 재해석할 수 있네"라고 발견한 것이지, 모노이드 자체는 범주론보다 먼저 존재했던 대수적 개념임.

---

## 모노이드 상세 설명

### 정의

모노이드는 세 가지가 필요함:

1. **집합**: 원소들의 모임
2. **이항 연산**: 원소 두 개를 받아서 같은 집합의 원소 하나를 내놓는 규칙
3. **두 가지 성질**: 결합법칙 + 항등원 존재

### 자연수 덧셈으로 풀어보면

집합: {0, 1, 2, 3, ...}
이항 연산: 덧셈(+)

**이항 연산이란**: "원소 두 개를 넣으면 같은 집합의 원소가 나온다." 3 + 5 = 8. 자연수 두 개를 넣었더니 자연수가 나옴. 집합 밖으로 튀어나가지 않음(닫힘 성질).

**결합법칙**: (2 + 3) + 4 = 2 + (3 + 4) = 9. 어디를 먼저 묶어서 계산하든 결과가 같음.

**항등원**: 0. 어떤 수에 0을 더해도 그 수 자체. 5 + 0 = 5, 0 + 5 = 5. 연산을 해도 상대방을 변화시키지 않는 원소.

이 세 조건을 만족하니까 (자연수, +, 0)은 모노이드.

### 프로그래밍 예시들

**문자열 + 이어붙이기(concat)**
- 집합: 모든 문자열
- 이항 연산: "hello" + "world" = "helloworld"
- 결합법칙: ("a" + "b") + "c" = "a" + ("b" + "c") = "abc"
- 항등원: 빈 문자열 "". "hello" + "" = "hello"

**리스트 + 이어붙이기**
- 집합: 모든 리스트
- 이항 연산: [1,2] + [3,4] = [1,2,3,4]
- 결합법칙: 리스트 세 개를 이어붙일 때 순서 무관
- 항등원: 빈 리스트 []

**정수 + 곱셈**
- 이항 연산: 3 × 5 = 15
- 결합법칙: (2 × 3) × 4 = 2 × (3 × 4) = 24
- 항등원: 1. 어떤 수에 1을 곱해도 그 수 자체.

### 모노이드가 강력한 이유

모노이드 구조가 있으면 **Fold(Reduce)**가 가능함. 리스트 [a, b, c, d]를 하나의 값으로 줄이는 것:

```
a ⊕ b ⊕ c ⊕ d
```

결합법칙이 있으니까 이걸 어떻게 쪼개든 결과가 같음:

```
(a ⊕ b) ⊕ (c ⊕ d)  -- 이렇게 둘씩 나눠서 병렬 처리 가능
```

이게 동시성 프로그래밍에서 중요한 이유임. 방대한 데이터를 여러 스레드에 나눠서 각각 계산한 후 결과를 합쳐도, 결합법칙 덕분에 순서에 상관없이 같은 결과가 보장됨. MapReduce의 Reduce가 정확히 이 원리임.

항등원은 "아무것도 없는 상태"의 기본값 역할. 빈 리스트를 fold하면? 항등원을 반환. 에러 없이 안전하게 처리됨.

## 군(Group)과 모노이드의 구분

역원이 필요 없는 상황에서 굳이 역원을 요구하지 않아도 되기 때문.

**역원이 있으면 군**: "되돌리기"가 가능. 정수 덧셈에서 +3을 했으면 -3으로 원래 상태 복구 가능.

**역원이 없으면 모노이드**: "되돌리기"가 불가능하지만 그래도 결합·축소는 가능.

현실에서 역원이 없는 경우가 훨씬 많음:

- 문자열 이어붙이기: "hello" + "world"를 했다고 "world"를 빼서 "hello"로 돌아갈 수 있나? 일반적인 concat 연산에 역원은 정의되지 않음.
- 로그 수집: 로그를 쌓기만 하지 되돌리지 않음.
- max 연산: max(3, 5) = 5인데, 5에서 3을 "되돌려서" 복구하는 건 불가능.

만약 군만 인정하고 모노이드를 구분하지 않으면, 이런 구조들에 대해 "결합법칙 + 항등원"이라는 유용한 성질을 활용할 수 없게 됨. 필요한 만큼만 요구해서 적용 범위를 최대화하는 것.

---

## 군의 실전 적용: Undo/Redo 시스템

군의 역원이 undo, 역원의 역원이 redo에 대응함.

텍스트 에디터의 편집 이력이 대표적인 예시:

- 연산: "3번 위치에 'hello' 삽입"
- 역원(undo): "3번 위치에서 'hello' 삭제"
- 역원의 역원(redo): 다시 "3번 위치에 'hello' 삽입"

항등원은 "아무것도 안 한 상태", 합성은 편집 연산을 이어붙이는 것, 결합법칙은 여러 편집을 어떤 순서로 묶든 결과가 같다는 보장.

실제로 Operational Transformation(Google Docs의 동시 편집)이나 CRDT 같은 분산 시스템의 동시성 제어 알고리즘이 이 군 구조를 활용함. 각 사용자의 편집을 연산으로 보고, 충돌 시 역원을 적용해서 되돌린 후 다시 합성하는 방식.

다만 모든 undo/redo가 엄밀한 군이 되려면 모든 연산에 대해 역원이 존재하고 합성이 닫혀있어야 하는데, 현실에서는 비가역적인 연산(파일 삭제 후 디스크에서 물리적 제거 등)도 있어서 완전한 군이 아닌 경우도 많음. 그런 경우는 모노이드 수준에서 다루게 됨.

---

## 모노이드에서 모나드로

### 모노이드 복습

자연수 덧셈 모노이드를 범주로 봤을 때:
- 대상: ★ 하나
- 사상: 0, 1, 2, 3, ... (자연수)
- 합성: 덧셈 (3 ∘ 2 = 5)
- 항등 사상: 0

### 모나드로 가는 과정

이제 "자기 함자(Endofunctor)들의 범주"를 생각함. 프로그래밍에서 자기 함자란 `List`, `Option`, `Future` 같은, 타입을 받아서 타입을 반환하는 것들.

이 함자들 사이에서 모노이드 구조를 찾으면:

| | 모노이드 (자연수 덧셈) | 모나드 |
|---|---|---|
| 집합의 원소 | 자연수 (0, 1, 2, ...) | 자기 함자 (List, Option, ...) |
| 이항 연산 | 덧셈 (+) | 함자의 합성 (∘) |
| 항등원 | 0 | 항등 함자 (Id) |

`List`를 두 번 적용하면 `List[List[A]]`가 됨. 이게 함자의 합성 `List ∘ List`.

### 문제 발생

자연수 덧셈에서 3 + 5 = 8이면 깔끔하게 하나의 자연수가 나옴. 그런데 함자의 합성 `List ∘ List`는 `List[List[A]]`라는 **중첩된 구조**가 됨. 이걸 다시 `List[A]`로 납작하게 만들어야 모노이드의 이항 연산처럼 "같은 집합 안에서 닫히는" 성질을 가질 수 있음.

이 "납작하게 만드는" 연산이 바로 **join (flatten)**:

$$\mu: T \circ T \to T$$

`List[List[A]]` → `List[A]`

그리고 항등원에 대응하는 것이 **unit (return, pure)**:

$$\eta: Id \to T$$

`A` → `List[A]` (평범한 값을 함자 안에 넣기)

### 모나드 법칙 = 모노이드 법칙

모노이드의 결합법칙과 항등원 법칙이 그대로 모나드 법칙이 됨:

**결합법칙**: 세 겹을 납작하게 만들 때, 바깥부터 펴든 안쪽부터 펴든 결과가 같아야 함.

$$\mu \circ T\mu = \mu \circ \mu T$$

`List[List[List[A]]]`를 `List[A]`로 만들 때, 안쪽 두 겹을 먼저 펴든 바깥 두 겹을 먼저 펴든 동일.

**항등법칙**: 값을 넣었다가 바로 펴면 원래와 같아야 함.

$$\mu \circ T\eta = \mu \circ \eta T = id_T$$

`A` → `List[A]` → `List[A]` (넣었다 펴면 그대로)

### 그래서 그 유명한 문장

> "모나드는 자기 함자 범주에서의 모노이드이다"

이 말은: **모노이드 구조에서 "원소"를 "자기 함자"로, "이항 연산"을 "함자의 합성 + join"으로, "항등원"을 "unit"으로 치환한 것**이 모나드라는 뜻. 패턴은 완전히 동일하고, 무엇을 그 자리에 넣느냐만 다름.

---

## 모노이드에서 모나드까지 필요한 모든 개념

### 1단계: 함자(Functor)

함자는 **범주와 범주 사이의 변환**임. 범주 $\mathcal{C}$에서 범주 $\mathcal{D}$로 가는 함자 $F$는 두 가지를 동시에 변환함:

1. 대상을 대상으로: $A \mapsto F(A)$
2. 사상을 사상으로: $f: A \to B \mapsto F(f): F(A) \to F(B)$

핵심은 변환 과정에서 범주의 구조(합성과 항등)가 보존된다는 것:

$$F(g \circ f) = F(g) \circ F(f)$$
$$F(id_A) = id_{F(A)}$$

프로그래밍에서 함자는 `List`, `Option`, `Future` 같은 **타입 생성자(Type Constructor)**에 대응됨.

- 대상의 변환: `Int` → `List[Int]` (타입을 감싸는 것)
- 사상의 변환: 함수 `f: A → B`를 `List[A] → List[B]`로 바꾸는 것

이 사상의 변환이 바로 **map** 함수임:

```python
# f: int → str 라는 사상이 있으면
f = lambda x: str(x)

# 함자(List)가 이걸 List[int] → List[str] 로 변환
list(map(f, [1, 2, 3]))  # ["1", "2", "3"]
```

### 2단계: 자기 함자(Endofunctor)

자기 함자는 출발 범주와 도착 범주가 **같은** 함자임. $F: \mathcal{C} \to \mathcal{C}$.

프로그래밍에서 거의 모든 함자가 자기 함자임. 왜냐하면 모든 변환이 "타입들의 범주"라는 단일한 범주 안에서 일어나기 때문. `Int` → `List[Int]`는 타입에서 타입으로 가는 변환이고, 출발지와 도착지 모두 같은 범주(Hask).

### 3단계: 자연 변환(Natural Transformation)

자연 변환은 **함자와 함자 사이의 변환**임.

함자 $F$와 함자 $G$가 있을 때, 자연 변환 $\alpha: F \Rightarrow G$는 모든 대상 $A$에 대해 사상 $\alpha_A: F(A) \to G(A)$를 제공하되, 어떤 사상 $f: A \to B$에 대해서든 다음이 성립해야 함:

$$G(f) \circ \alpha_A = \alpha_B \circ F(f)$$

이건 "변환의 순서를 바꿔도 결과가 같다"는 뜻.

```python
# headOption: List[A] → Option[A] 는 자연 변환
# 어떤 함수 f를 먼저 map하고 headOption을 하든,
# headOption을 먼저 하고 f를 map하든 결과가 같아야 함
head_option(list(map(f, xs))) == map(f, head_option(xs))
```

### 4단계: 함자의 합성

함자도 합성할 수 있음. 자기 함자를 자기 자신과 합성하면 "상자 안에 상자"가 생김:

```
T(A)        = List[A]
T(T(A))     = List[List[A]]
T(T(T(A)))  = List[List[List[A]]]
```

### 5단계: 자기 함자들의 범주

자기 함자 자체를 대상으로 하는 새로운 범주를 구성:

| | 기존 범주 (Hask) | 새로운 범주 (자기 함자의 범주) |
|---|---|---|
| 대상 | 타입 | 자기 함자 (List, Option, ...) |
| 사상 | 함수 | 자연 변환 |
| 합성 | 함수 합성 | 자연 변환의 합성 |
| 항등 사상 | identity 함수 | 항등 자연 변환 |

### 6단계: 모노이드 구조를 자기 함자에 적용 → 모나드

모노이드의 세 조건을 자기 함자 세계로 번역:

**이항 연산 → join (μ)**

$$\mu: T \circ T \Rightarrow T$$

```python
# List의 join
[[1, 2], [3, 4]]  →  [1, 2, 3, 4]

# Option의 join
Some(Some(5))     →  Some(5)
Some(None)        →  None
```

**항등원 → unit (η)**

$$\eta: Id \Rightarrow T$$

```python
# List의 unit
5  →  [5]

# Option의 unit
5  →  Some(5)
```

**결합법칙과 항등법칙 → 모나드 법칙**

결합법칙: 세 겹짜리를 한 겹으로 만들 때, 어느 두 겹을 먼저 합치든 결과가 같아야 함.

$$\mu \circ T\mu = \mu \circ \mu T$$

```python
[[[1, 2], [3]], [[4]]]

# 안쪽 먼저 flatten → [[1, 2, 3], [4]] → [1, 2, 3, 4]
# 바깥 먼저 flatten → [[1, 2], [3], [4]] → [1, 2, 3, 4]
# 결과 동일 ✅
```

항등법칙: unit으로 넣었다가 join으로 펴면 원래와 같아야 함.

$$\mu \circ T\eta = \mu \circ \eta T = id_T$$

```python
[1, 2, 3]
→ [[1], [2], [3]]            # 각 원소에 unit 적용
→ [1, 2, 3]                  # join 적용, 원래와 동일 ✅

[1, 2, 3]
→ [[1, 2, 3]]                # 전체에 unit 적용
→ [1, 2, 3]                  # join 적용, 원래와 동일 ✅
```

### 전체 그림

```
값      →  타입(대상)  →  함자(타입→타입)  →  자기 함자  →  모나드
               ↕              ↕                  ↕            ↕
          범주의 대상      범주 간 변환        같은 범주 안    모노이드 구조를
                                              에서의 변환    자기 함자에 적용
```

## 함자의 기원 — 범주론

함자는 범주론의 핵심 개념임. 범주론이 1940년대에 에일렌베르크와 맥레인에 의해 만들어졌는데, 사실 범주론을 만든 **원래 목적 자체가 함자를 정의하기 위해서**였음.

순서:

1. 함자라는 패턴이 여러 수학 분야에서 반복됨을 발견
2. 이걸 정의하려면 "범주"라는 개념이 먼저 필요
3. 범주를 정의함
4. 함자를 정의함
5. 함자 사이의 변환(자연 변환)을 정의함

범주론은 함자를 낳기 위해 만들어진 이론이고, 자연 변환은 함자를 다루기 위해 만들어진 도구.

---

## Hask 상세 설명

Hask는 Haskell 프로그래밍 언어의 타입과 함수로 구성된 범주의 이름.

- **대상**: 모든 타입. `Int`, `String`, `Bool`, `List[Int]`, `Option[String]`, `(Int, Bool)` 등 전부.
- **사상**: 타입 사이의 함수. `length: String → Int`, `not: Bool → Bool` 등.
- **합성**: 함수 합성. `g ∘ f`
- **항등 사상**: `id: A → A` (identity 함수)

"Hask"라는 이름을 붙여서 부르는 이유는, 범주론에서 다루는 범주가 여러 개이기 때문에 구분이 필요해서임. Set, Ord, Vect 등과 구별하기 위해.

엄밀하게 말하면 Hask는 수학적으로 완벽한 범주가 아님. 무한루프(`undefined`)나 예외 같은 bottom value(⊥)가 존재하기 때문. 하지만 실용적으로는 범주로 취급하고 범주론의 도구들을 적용해도 문제가 없어서, 사실상 범주로 다룸.

## 자연 변환(Natural Transformation) 상세

### 직관: "모든 타입에 대해 일관되게 작동하는 변환"

함자 $F$와 함자 $G$가 있을 때, 자연 변환 $\alpha: F \Rightarrow G$는 "F라는 상자를 G라는 상자로 바꾸는 방법"인데, **상자 안에 뭐가 들어있든 상관없이 동일한 방식으로** 작동해야 함.

### 예제 1: `safeHead: List[A] → Option[A]`

```python
def safe_head(xs: list[A]) -> Option[A]:
    return Some(xs[0]) if xs else None
```

이건 `List` 함자에서 `Option` 함자로 가는 자연 변환임.

- `List[Int]` → `Option[Int]`: `[3, 1, 4]` → `Some(3)`
- `List[String]` → `Option[String]`: `["hello", "world"]` → `Some("hello")`
- `List[Bool]` → `Option[Bool]`: `[]` → `None`

안에 뭐가 들어있든 **"첫 번째 원소를 꺼내거나 없으면 None"**이라는 동일한 규칙으로 작동함.

### 자연성 조건(Naturality Condition)

$$G(f) \circ \alpha_A = \alpha_B \circ F(f)$$

`safeHead`로 풀어보면, 함수 `f: Int → String`이 `str()`이라고 할 때:

**경로 1**: 먼저 변환하고 나중에 map
```
[3, 1, 4]  →  safeHead  →  Some(3)  →  map(str)  →  Some("3")
```

**경로 2**: 먼저 map하고 나중에 변환
```
[3, 1, 4]  →  map(str)  →  ["3", "1", "4"]  →  safeHead  →  Some("3")
```

결과가 같음. **어느 경로로 가든 도착지가 동일**.

### 예제 2: `length: List[A] → Int`는 자연 변환이 아님

`Int`는 함자가 아님. 자연 변환은 **함자에서 함자로** 가는 것이어야 하므로 자연 변환이 아님.

### 예제 3: `unit: A → List[A]`

항등 함자 $Id$에서 $List$ 함자로 가는 자연 변환.

```
경로 1: 5  →  unit  →  [5]  →  map(str)  →  ["5"]
경로 2: 5  →  str   →  "5"  →  unit      →  ["5"]
```
동일. ✅

### 예제 4: `join: List[List[A]] → List[A]`

$List \circ List$ 함자에서 $List$ 함자로 가는 자연 변환.

```
경로 1: [[1,2],[3]]  →  join  →  [1,2,3]  →  map(str)  →  ["1","2","3"]
경로 2: [[1,2],[3]]  →  map(map(str))  →  [["1","2"],["3"]]  →  join  →  ["1","2","3"]
```
동일. ✅

### 자연 변환이 아닌 것: 내용물을 들여다보는 변환

```python
def weird(xs: list[int]) -> Option[int]:
    return Some(xs[0] + 1) if xs else None  # +1을 하고 있음
```

`Int`에만 작동하고, 안에 든 값의 내용을 직접 조작함. **모든 타입에 대해 일관되게** 작동하지 않으므로 자연 변환이 아님.

### 모나드와의 연결

모나드의 두 핵심 연산이 둘 다 자연 변환임:

| 연산 | 자연 변환 | 하는 일 |
|---|---|---|
| `unit` ($\eta$) | $Id \Rightarrow T$ | 값을 상자에 넣기 |
| `join` ($\mu$) | $T \circ T \Rightarrow T$ | 중첩된 상자를 펴기 |

## 프로그래밍에서의 함자 상세 예제

### Hask 범주에서의 Hom 집합

```
Hom(Int, String) = {str, hex, bin, lambda x: "hello" * x, ...}
Hom(Int, Int) = {identity, (+1), (*2), negate, ...}
Hom(Int, Bool) = {is_even, is_positive, ...}
```

각 Hom 집합에 원소가 무한히 많기 때문에 Thin category가 아님.

### List 함자

**대상의 변환:**
```
Int       →  List[Int]
String    →  List[String]
Bool      →  List[Bool]
```

**사상의 변환 (map):**

```python
f = str  # f: Int → String

# List 함자가 f를 변환: List[Int] → List[String]
list(map(f, [1, 2, 3]))  # ["1", "2", "3"]
```

```
원래 세계:    Int  ---f--->  String
              ↓                ↓
List 함자:  List[Int] --map(f)--> List[String]
```

**함자 법칙 확인:**

```python
# 1. 합성 보존: map(g ∘ f) = map(g) ∘ map(f)
f = str
g = len
xs = [1, 2, 3]

list(map(lambda x: g(f(x)), xs))        # [1, 1, 1]
list(map(g, list(map(f, xs))))          # [1, 1, 1]
# 동일 ✅

# 2. 항등 보존: map(id) = id
list(map(lambda x: x, [1, 2, 3]))      # [1, 2, 3]
# 원래 리스트와 동일 ✅
```

### Option(Maybe) 함자

```python
def option_map(f, opt):
    match opt:
        case Some(value):
            return Some(f(value))
        case Nothing():
            return Nothing()

option_map(str, Some(42))    # Some("42")
option_map(str, Nothing())   # Nothing()
```

`Nothing`일 때는 함수를 적용하지 않고 그대로 통과시킴.

### 함자가 아닌 것

```python
def broken_map(f, xs):
    return list(reversed(map(f, xs)))
```

`reverse`가 끼어들면서 합성 보존이 깨짐. map을 한 번 하면 뒤집히고, 두 번 하면 다시 돌아오기 때문에 `map(g ∘ f) ≠ map(g) ∘ map(f)`. 함자 법칙을 만족하지 않으므로 함자가 아님.

### 함자의 본질 정리

| 함자 | 씌우는 문맥 | map이 하는 일 |
|---|---|---|
| `List` | "여러 개일 수 있음" | 모든 원소에 함수 적용 |
| `Option` | "없을 수도 있음" | 있으면 적용, 없으면 통과 |
| `Future` | "아직 안 왔음" | 값이 도착하면 적용 |
| `Dict[K, _]` | "키와 연관됨" | 모든 값에 함수 적용 |
| `Tree` | "트리 구조" | 모든 노드에 함수 적용 |

## 모노이드 구조는 Hask가 아닌 상위 범주에서 등장

Hask 안에서 직접 등장하는 게 아님. **Hask 위에 한 단계 더 올라간 범주**에서 등장함.

**1층: Hask** — 대상이 여러 개(타입들). 모노이드가 아님.

**2층: End(Hask) (자기 함자들의 범주)** — 대상이 여러 개(함자들). 전체로서는 모노이드가 아님.

**2층에서 특정 함자 하나를 골라서** 모노이드 구조를 봄:

| 모노이드 조건 | 자연수 덧셈 | List 모나드 |
|---|---|---|
| 원소들 | 0, 1, 2, 3, ... | Id, List, List ∘ List, ... |
| 이항 연산 | + | 합성 후 join으로 납작하게 |
| 항등원 | 0 | Id (unit η로 연결) |
| 결합법칙 | (a+b)+c = a+(b+c) | 세 겹을 어디서 먼저 펴든 동일 |

```
Hask (타입과 함수)
  ↑ 한 단계 위
End(Hask) (자기 함자와 자연 변환)
  ↑ 여기서 특정 함자 T를 골라 모노이드 구조를 부여
모나드 = T + join(μ) + unit(η) + 모나드 법칙
```

## 여러 인자를 가지는 함수는 Hask에서 어떻게 다루는가

범주론에서 사상은 항상 "대상 하나 → 대상 하나". "인자 두 개를 받는 사상"은 없음.

### 방법 1: 곱 타입(Product Type)으로 묶기

```python
# 사상 하나로 변환: (Int, Int) → Int
def add(pair: tuple[int, int]) -> int:
    return pair[0] + pair[1]
```

`(Int, Int)`는 곱 타입이라는 하나의 대상.

### 방법 2: 커링(Currying)

```python
# Int → (Int → Int)
def add(x: int) -> Callable[[int], int]:
    return lambda y: x + y

add(3)(5)  # 8
```

`Int → Int`도 Hask의 대상(타입)이기 때문에 성립함.

커링이 가능한 이유는 Hask가 **닫힌 범주(Closed Category)**이기 때문. "함수 타입 $A \to B$가 그 자체로 범주의 대상이다"라는 성질.

---

## 닫힌 범주(Closed Category) 상세

"닫힘(Closed)"이라는 이름은 **연산의 결과가 같은 세계 안에 머무른다**는 뜻에서 옴.

### 닫혀있지 않은 경우

순서론의 범주(Ord)에서:

- 대상: 1, 2, 3, ...
- 사상: ≤ 관계

$Hom(2, 6) = \{2 \leq 6\}$인데, 이 집합 자체는 범주의 대상(숫자)이 아님. 사상들의 모임은 범주 **바깥**에서만 존재함.

### 닫힌 범주

Hask에서 `Int → String`이라는 **함수 타입** 자체가 Hask의 대상(타입)임:

```python
f: Callable[[int], str]          # 이것 자체가 타입
fs: list[Callable[[int], str]]   # 함수의 리스트도 가능
```

사상들의 모임이 범주 바깥으로 나가지 않고, 범주 **안의 대상**으로 존재함. 그래서 "닫혀있다(Closed)".

### 커링과의 연결

$B \to C$가 범주의 대상이 아니면 $f': A \to (B \to C)$라는 사상을 범주 안에서 쓸 수 없음. 도착지가 범주 바깥에 있으니까. 닫힌 범주에서는 $B \to C$가 대상이므로 정당한 사상.

### "닫힘"이라는 용어의 일관성

- 집합의 닫힘: 자연수가 덧셈에 대해 닫혀있다 = 결과가 자연수 밖으로 안 나감
- 모노이드의 닫힘: 이항 연산의 결과가 같은 집합 안에 머무름
- 범주의 닫힘: 사상들의 모임이 같은 범주 안에 대상으로 머무름

전부 "연산의 결과가 같은 세계 안에 있다"는 동일한 직관.

## 커링과 범주론의 연결 — 수반(Adjunction)

커링은 범주론에서 **곱(Product)과 지수(Exponential) 사이의 수반(Adjunction)**으로 기술됨.

$$Hom(A \times B, C) \cong Hom(A, C^B)$$

프로그래밍으로 번역:
```
(A, B) → C    ⟺    A → (B → C)
```

왼쪽은 인자 두 개를 동시에 받는 함수, 오른쪽은 커링된 함수. 이 둘이 **동형(isomorphic)**.

## Hom 집합, 곱 대상, 지수 대상의 엄밀한 정의

### Hom 집합

범주 $\mathcal{C}$에서 대상 $A$와 $B$가 있을 때, $A$에서 $B$로 가는 **모든 사상들의 모임**:

$$Hom_{\mathcal{C}}(A, B) \quad \text{또는} \quad \mathcal{C}(A, B)$$

범주 **바깥**에 존재하는 집합. 범주의 대상이 아님.

### 곱 대상 (Product)

대상 $A$와 $B$의 곱 $A \times B$란, 투영 사상 $\pi_1: P \to A$, $\pi_2: P \to B$가 있어서:

**보편 성질:** 임의의 대상 $X$와 사상 $f: X \to A$, $g: X \to B$가 있을 때, 다음을 만족하는 **유일한** 사상 $\langle f, g \rangle: X \to P$가 존재:

$$\pi_1 \circ \langle f, g \rangle = f$$
$$\pi_2 \circ \langle f, g \rangle = g$$

프로그래밍에서의 대응:

```python
# 곱 대상 = 튜플 타입
P = tuple[int, str]

# 투영 사상
pi_1 = lambda p: p[0]
pi_2 = lambda p: p[1]

# 보편 성질
f = lambda x: x + 1
g = lambda x: str(x)
pair = lambda x: (f(x), g(x))  # ⟨f,g⟩

pi_1(pair(5))  # 6 == f(5) ✅
pi_2(pair(5))  # "5" == g(5) ✅
```

### 지수 대상 (Exponential)

대상 $B$와 $C$의 지수 대상 $C^B$란, 평가 사상 $eval: E \times B \to C$가 있어서:

**보편 성질:** 임의의 대상 $A$와 사상 $f: A \times B \to C$가 있을 때, 다음을 만족하는 **유일한** 사상 $\tilde{f}: A \to E$가 존재:

$$eval \circ (\tilde{f} \times id_B) = f$$

이 $\tilde{f}$가 바로 커링된 형태.

$eval$은 "함수와 인자를 받아서 적용하는" 사상:

```python
def eval(pair: tuple[Callable[[int], str], int]) -> str:
    f, x = pair
    return f(x)

eval((str, 42))  # "42"
```

### 세 개념의 관계

| | Hom 집합 | 지수 대상 |
|---|---|---|
| 존재 위치 | 범주 바깥 (메타 수준) | 범주 안 (대상) |
| 표기 | $Hom(A, B)$ | $B^A$ |
| 항상 존재? | 모든 범주에서 존재 | 닫힌 범주에서만 존재 |

닫힌 범주에서는 $Hom(A, B)$와 $B^A$가 자연스럽게 대응. 외부에서 볼 수 있던 사상들의 모임이, 범주 내부의 대상으로도 존재.

곱 대상과 지수 대상이 이루는 동형:

$$Hom(A \times B, C) \cong Hom(A, C^B)$$

이것이 커링.

## Hom 집합의 이중성과 모노이드 곱

### "사상이면서 동시에 집합의 원소"

대상이 하나($m$)뿐인 범주 $M$에서 모든 사상이 $m \to m$임. 이 사상들을 모아놓은 것이 Hom 집합 $M(m, m)$.

이중적인 성격:

- **사상으로서**: 다른 사상과 **합성**할 수 있음 ($g \circ f$)
- **집합의 원소로서**: 집합 $M(m, m)$ 안에 들어있는 **원소**임

### "모노이드 곱"이란

모노이드의 이항 연산을 "모노이드 곱(monoid multiplication)"이라고 부르는 것. 대수학에서 모노이드의 이항 연산을 전통적으로 곱셈 표기($a \cdot b$)로 쓰는 관습 때문.

자연수 덧셈 모노이드를 대상 하나짜리 범주로 봤을 때:

사상들의 **합성**(범주론 관점): $3 \circ 2 = 5$

같은 것을 **집합의 원소 간 모노이드 곱**(대수학 관점): $3 + 2 = 5$

같은 연산인데, 범주론에서는 "사상의 합성"이라 부르고, 대수학에서는 "모노이드 곱"이라 부르는 것.

Hom 집합의 원소들이 단순한 원소가 아니라 합성 가능한 사상이기 때문에, **집합론적 구조(모노이드)와 범주론적 구조(사상의 합성)가 자연스럽게 연결**됨.

## 모나드의 기원 — 범주론

모나드는 범주론에서 나온 순수 수학 개념임.

1958년에 로저 고드망(Roger Godement)이 대수위상수학에서의 층(sheaf) 이론을 연구하면서 처음 정의했고, 이후 1960년대에 맥레인과 에일렌베르크 등이 체계화. 원래 이름은 "triple" 또는 "standard construction"이었고, 나중에 "monad"로 굳어짐.

컴퓨터 과학으로 가져온 건 **에우제니오 모지(Eugenio Moggi)**가 1989~1991년에 발표한 논문에서. 그리고 **필립 와들러(Philip Wadler)**가 1990년대 초 Haskell에 실제로 구현하면서 프로그래밍 언어에 정착.

순서:

1. **1958~1960년대**: 범주론에서 순수 수학 개념으로 탄생
2. **1989~1991**: 모지가 컴퓨터 과학에 적용 가능함을 이론적으로 증명
3. **1990년대 초**: 와들러가 Haskell에 구현하여 실용화

수학에서 30년 넘게 존재하던 개념을 컴퓨터 과학이 가져다 쓴 것.
