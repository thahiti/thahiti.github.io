# 데이터 파이프라인의 진화: ETL의 기원에서 dbt, 온톨로지 기반 AI 아키텍처까지

데이터 엔지니어링에 발을 들이면 ETL, ELT, dbt, 시맨틱 레이어 같은 용어를 매일같이 마주치게 됩니다. 그런데 막상 이런 질문 앞에서는 말문이 막힙니다.

> *"ETL은 언제, 왜 생겨났을까? 추출·변환·적재의 순서가 ELT로 뒤집힌 이유는 무엇일까? dbt는 도대체 무엇을 해결하려고 등장했고, 어쩌다 이렇게까지 표준이 되었을까?"*

오늘날 우리가 '모던 데이터 스택(Modern Data Stack)'이라 부르는 클라우드 네이티브 데이터 생태계는 하루아침에 만들어진 것이 아닙니다. 분절된 트랜잭션 시스템(OLTP)에 흩어진 데이터를 통합해 분석적 가치를 끌어내려 했던 산업계의 수십 년에 걸친 고군분투, 그리고 그 위에 누적된 컴퓨팅 패러다임의 변화가 켜켜이 쌓인 결과물입니다.

데이터를 원천 시스템에서 추출(Extract)하고, 비즈니스 요구사항에 맞게 변환(Transform)하여, 중앙 저장소에 적재(Load)하는 일련의 과정인 ETL은 초기 데이터 웨어하우스(Data Warehouse)의 등장과 함께 태어났습니다. 이후 클라우드 컴퓨팅 환경이 도래하면서 그 순서가 ELT로 뒤집혔고, 이 흐름 속에서 분석가에게 소프트웨어 엔지니어링의 엄격함을 쥐여 준 dbt(data build tool)가 등장해 '분석 엔지니어링(Analytics Engineering)'이라는 새로운 직군을 열었습니다.

그리고 파이프라인이 성숙기에 접어든 지금, 데이터 생태계는 새로운 과제와 마주하고 있습니다. 데이터를 깨끗하게 정제해 저장하는 것을 넘어, 비즈니스 용어의 의미를 전사적으로 통일하고, 거대 언어 모델(LLM)과 AI 에이전트가 데이터를 오해 없이 추론할 수 있도록 문맥(Context)을 제공해야 한다는 요구입니다. 이 지점에서 데이터를 수치로 집계하는 시맨틱 레이어(Semantic Layer)와, 데이터의 의미와 관계를 기계가 읽을 수 있도록 정의하는 온톨로지(Ontology)의 결합이 차세대 AI 준비형 아키텍처의 핵심으로 떠오르고 있습니다.

이 글에서는 ETL이라는 개념이 언제, 왜 생겨났는지 그 기원을 1960년대부터 추적하고, 인프라의 변화에 따라 파이프라인이 어떻게 진화했는지 살펴봅니다. 이어 dbt가 어떤 조직적·기술적 요구에서 탄생했는지, 그리고 시맨틱 레이어와 온톨로지가 어떻게 맞물려 차세대 데이터 프레임워크로 엮이는지를 단계적으로 정리하겠습니다.

## 1. 데이터 통합의 태동: 1960년대부터 1980년대까지

### 1.1 물리적 장부에서 전자 스토리지로

데이터를 통합해 비즈니스 통찰을 얻으려는 요구는 현대적 의미의 관계형 데이터베이스가 보급되기 훨씬 전부터 존재했습니다. 1960년대는 기업의 데이터 관리가 물리적 장부와 천공 카드(Punch card)에서 벗어나 자기 테이프(Magnetic tape)와 초기 형태의 디스크 스토리지로 옮겨 가던 시기였습니다. 자기 테이프는 순차적인 대용량 데이터의 일괄 처리(Batch processing)를 가능하게 했고, 디스크 스토리지는 무작위 접근(Random access)과 데이터 처리의 유연성을 제공했습니다.

이러한 하드웨어의 발전은 학계와 산업계가 데이터 분석의 이론적 틀을 마련하는 계기가 되었습니다. 1960년대 General Mills와 Dartmouth College의 공동 연구를 통해, 현대 차원 모델링(Dimensional Modeling)의 근간이 되는 '차원(Dimensions)'과 '팩트(Facts)'라는 용어가 처음으로 정립되었습니다. 1970년대에 접어들면서 ACNielsen, IRI 같은 시장 조사 기관들은 소매 판매 데이터 분석을 위해 이 개념을 적용한 초보적인 형태의 차원 데이터 마트(Data Mart)를 시장에 공급하기 시작했습니다.

이 시기 기업들은 부서별로 서로 다른 애플리케이션과 트랜잭션 시스템을 운영하고 있었고, 이는 심각한 데이터 사일로(Silo) 현상을 낳았습니다. 인사부, 영업부, 구매부의 데이터가 물리적·논리적으로 분리되어 있어, 전사적인 비용·수익 분석을 수행하려면 막대한 수작업과 중복된 데이터 복사가 필요했습니다. 이러한 비효율을 해결하기 위해 1970년대 중반, 훗날 '데이터 웨어하우스의 아버지'라 불리게 되는 빌 인몬(Bill Inmon)은 기업 내 여러 시스템에 흩어진 데이터를 한곳에 모아 주제 지향적(Subject-oriented)이고 통합된 형태로 저장하는 '데이터 웨어하우스'라는 개념을 정의하고 논의하기 시작했습니다.

초기 하드웨어·소프트웨어 벤더들도 이러한 통합 분석의 요구에 부응하는 시스템을 개발하기 시작했습니다. 1975년 Sperry Univac은 세계 최초의 4세대 프로그래밍 언어(4GL)를 포함한 데이터베이스 관리·보고 시스템 MAPPER(Maintain, Prepare, and Produce Executive Reports)를 선보였습니다. 이는 현대적인 엔터프라이즈 데이터 웨어하우징 플랫폼의 전신인 '정보 센터(Information Centers)'를 구축하기 위해 설계된 최초의 플랫폼이었습니다.

### 1.2 비즈니스 데이터 웨어하우스 개념의 공식화

1980년대는 관계형 데이터베이스의 발전과 맞물려, 데이터 통합 아키텍처가 이론적 수준을 넘어 실제 기업의 핵심 인프라로 자리 잡기 시작한 시기였습니다. 데이터의 양이 기하급수적으로 늘면서 범용 컴퓨터로는 복잡한 분석 쿼리를 처리하기 어려워졌고, 1983년 Teradata는 오직 의사결정 지원(Decision support)만을 목적으로 설계된 특수 목적 데이터베이스 컴퓨터 DBC/1012를 출시했습니다. 1984년에는 David Liddle과 Don Massaro가 설립한 Metaphor Computer Systems가, 비즈니스 사용자들이 직접 데이터를 분석할 수 있도록 지원하는 하드웨어·소프트웨어·그래픽 사용자 인터페이스(GUI) 패키지를 발표했습니다.

이러한 기술적 흐름 속에서, 분산된 운영 시스템(Operational Systems)으로부터 의사결정 지원 환경으로 데이터가 흘러가는 과정을 체계화하려는 시도가 이어졌습니다. 1988년 IBM의 연구원 Barry Devlin과 Paul Murphy는 《IBM Systems Journal》에 "An architecture for a business and information system"이라는 논문을 발표하며 "비즈니스 데이터 웨어하우스(Business Data Warehouse)"라는 용어를 공식적으로 소개했습니다. 이들은 여러 의사결정 지원 환경을 각각 독립적으로 지원하기 위해 레거시 운영 시스템에서 데이터를 따로따로 수집·정제·통합할 때 발생하는 막대한 중복과 비용 문제를 지적하며, 이를 해결할 중앙 집중형 아키텍처 모델을 제안했습니다.

기업의 인식이 근본적으로 바뀐 또 다른 계기는 1991년 출간된 James M. Kerr의 저서 《The IRM Imperative》였습니다. 이 책은 조직의 데이터 자원을 효과적으로 관리하고, 그 가치를 금전적으로 환산해 재무상태표의 자산으로 보고하자는 아이디어를 대중화했습니다. Kerr는 트랜잭션 주도형 시스템에서 파생된 데이터로 주제 영역별 데이터베이스를 구축하고, 이 요약 데이터를 경영진의 의사결정에 활용하는 구체적인 방법을 서술했습니다. 이는 데이터 웨어하우스가 단순한 IT 부서의 저장 공간이 아니라 기업의 경쟁력을 좌우하는 전략적 자산이라는 인식을 확산시키는 데 크게 기여했습니다.

| 연대 | 주요 발전 사항 | 관련 인물·기업 |
|---|---|---|
| 1960년대 | 천공 카드에서 자기 테이프·디스크 스토리지로 전환. '차원(Dimension)'과 '팩트(Fact)' 용어 등장 | General Mills, Dartmouth College |
| 1970년대 | 소매 판매용 차원 데이터 마트 등장. 데이터 웨어하우스 개념 논의 시작. MAPPER 출시 | ACNielsen, IRI, Bill Inmon, Sperry Univac |
| 1980년대 | 의사결정 지원 전용 데이터베이스·분석 GUI 패키지 등장. '비즈니스 데이터 웨어하우스' 용어·아키텍처 공식화 | Teradata, Metaphor Computer Systems, Barry Devlin, Paul Murphy |

## 2. ETL의 탄생과 산업 표준화: 1990년대의 데이터 파이프라인

데이터 웨어하우스라는 목적지의 구조가 구체화되자, 이번에는 데이터를 그곳으로 옮기기 위한 논리적·물리적 파이프라인의 필요성이 대두되었습니다. 운영 시스템의 데이터는 데이터 웨어하우스의 엄격한 분석 구조와 맞지 않았고, 중복 제거·형식 통일·결측치 처리 등 품질을 확보하는 과정이 필수적이었습니다. 1980년대 후반까지도 이런 데이터 연동 작업은 엔지니어들이 데이터 소스별로 맞춤형 스크립트(Custom scripts)를 일일이 손으로 작성해 수행했습니다. 이러한 수동 접근은 노동 집약적이었을 뿐 아니라, 잦은 오류와 병목을 유발해 데이터 관리 워크플로 전반의 민첩성을 떨어뜨렸습니다.

### 2.1 상용 ETL 도구의 등장과 개념의 정립

이러한 시장의 고충을 해결하고 자동화된 파이프라인 구축을 지원하기 위해, 1990년대 초반 ETL(Extract, Transform, Load)이 하나의 독립된 소프트웨어 카테고리로 확립되기 시작했습니다. 이는 데이터를 원천에서 '추출'하고, 별도의 스테이징 환경에서 일관된 포맷으로 '변환'·정제한 뒤, 데이터 웨어하우스에 '적재'하는 세 단계의 컴퓨팅 프로세스를 의미합니다.

ETL이라는 용어와 상용 제품의 확산에 결정적인 역할을 한 인물은, 앞서 데이터 웨어하우스 개념을 정립했던 빌 인몬입니다. 인몬은 1991년 Prism Solutions라는 기업을 설립하고, 업계 최초로 데이터 웨어하우스 개발과 데이터 이동을 관리하는 소프트웨어 'Prism Warehouse Manager'를 선보였습니다. 수작업 코딩에 의존하던 데이터 통합 과정을 상용 소프트웨어로 자동화하려는, 산업계 최초의 의미 있는 시도였습니다.

이어 1992년 빌 인몬은 자신의 철학을 집대성한 저서 《Building the Data Warehouse》를 출간해, 전사 데이터를 중앙 집중화하고 제3정규형(3NF)으로 정규화해 저장하는 하향식(Top-down) 엔터프라이즈 데이터 웨어하우스 설계 방법론을 공식화했습니다. 반면 1996년 데이터베이스 선구자 랄프 킴볼(Ralph Kimball)은 《The Data Warehouse Toolkit》을 출간하며 인몬의 정규화 모델에 반기를 들었습니다. 킴볼은 부서 단위의 특정 비즈니스 프로세스 분석에 최적화된 차원 모델링(스타 스키마) 기반 데이터 마트를 먼저 구축하고 이를 결합해 나가는 상향식(Bottom-up) 설계 방법론을 제시했습니다.

두 거장의 철학적 논쟁(Inmon vs Kimball)은 데이터 웨어하우징 설계 방법론을 성숙시켰습니다. 그리고 두 모델 모두 데이터를 목적지에 맞게 극도로 정제하고 변환해야 한다는 공통된 요구를 안고 있었기에, ETL 인프라 시장의 성장을 폭발적으로 견인하는 촉매가 되었습니다.

### 2.2 1세대 ETL 도구의 전성기와 메타데이터 폐쇄성의 한계

1990년대 후반부터 2000년대 초반에 이르러 Informatica PowerMart(이후 PowerCenter), IBM DataStage, Oracle Data Integrator(과거 Sunopsis) 같은 목적형(Purpose-built) 1세대 상용 ETL 도구들이 시장을 장악했습니다. 이들의 가장 큰 특징은 복잡한 데이터 변환 로직을 그래픽 사용자 인터페이스(GUI)의 드래그 앤 드롭 방식으로 구현할 수 있게 했다는 점입니다. 이는 개발 시간을 단축하고 데이터 통합 과정을 시각화하는 데 기여했지만, 치명적인 구조적 한계를 안고 있었습니다.

당시 대부분의 개발팀은 ETL 메타데이터와 파이프라인 설정이 소프트웨어 내부의 독점적인(Proprietary) 저장소에 갇혀 있는 폐쇄적 환경에서 작업해야 했습니다. 1990년대 후반의 Informatica PowerMart 4.x 같은 초기 버전은 작업 내역을 사람이 읽을 수 있는 텍스트 형식으로 내보내는 기능이 극히 제한적이었습니다. 따라서 파이프라인의 버전 관리, 외부 스크립트를 통한 자동화, CI/CD(지속적 통합·배포) 같은 소프트웨어 엔지니어링의 모범 사례를 데이터 엔지니어링에 적용하는 것은 사실상 불가능했습니다.

이러한 폐쇄성은 2001년 무렵 Informatica PowerCenter 5.1 버전에서 저장소 객체를 외부 환경으로 마이그레이션할 수 있는 XML 기반 내보내기·가져오기 기능이 도입되고, IBM DataStage가 DSX 파일 형식(구조화된 텍스트 형식)을 지원하면서 일부 완화되었습니다. Oracle ODI 역시 메타데이터를 관계형 저장소에 보관하면서도 XML 형태의 추출을 지원해, 환경 간 구성 이동과 버전 관리의 가능성을 열었습니다.

그럼에도 대다수의 기업 데이터 팀은 외부 스크립트 기반의 자동화를 적극적으로 받아들이지 않고, 여전히 수작업 중심의 GUI 인터페이스에 머무는 길을 택했습니다. 이 시기 데이터 파이프라인이 반드시 'ETL(추출 → 변환 → 적재)'의 순서를 엄격히 따라야 했던 근본적인 이유는, 당시 목적지 시스템(온프레미스 기반의 관계형 데이터베이스와 하드웨어 어플라이언스)의 물리적·재무적 한계 때문이었습니다. 스토리지 비용은 매우 비쌌고 컴퓨팅 자원은 제한적이었으므로, 웨어하우스의 스키마에 맞지 않는 불필요한 원시(Raw) 데이터를 최종 목적지까지 끌고 들어가는 것은 인프라 낭비이자 쿼리 성능의 심각한 저하를 의미했습니다. 그래서 데이터를 적재하기 전, 완전히 분리된 전용 변환 서버(Dedicated transformation layer)나 중간 스테이징(Staging) 환경에서 무거운 연산을 수행해 데이터를 철저히 정제하고 비즈니스 규칙을 적용해야만 했습니다.

## 3. 클라우드 컴퓨팅과 패러다임 전환: ELT의 부상과 실시간 처리

2000년대 후반에 접어들며 인터넷 인프라의 확충, 스마트폰의 보급, 소셜 미디어와 사물인터넷(IoT) 센서의 확산은 이른바 '빅데이터(Big Data)' 시대를 촉발했습니다. 데이터의 총량은 과거와 비교할 수 없을 만큼 팽창했고, 형태 또한 엄격한 테이블 구조의 정형 데이터를 넘어 JSON, XML, 애플리케이션 로그, 웹 스크래핑 텍스트 등 반정형(Semi-structured)·비정형(Unstructured) 데이터로 다변화되었습니다.

기존의 온프레미스 전용 서버 기반 ETL 파이프라인은 이러한 막대한 데이터의 볼륨(Volume), 속도(Velocity), 다양성(Variety)을 처리하는 데 심각한 병목을 겪었습니다. 데이터가 변환 서버를 거치는 동안 처리 지연이 발생했고, 야간 배치(Nightly batch) 작업에 의존하는 구조 탓에 현업 분석가들은 실시간 상황을 반영하지 못하는 과거의 데이터에 의존해야 했습니다. 게다가 변환 과정에서 원시 데이터가 폐기되고 정제된 결과물만 웨어하우스에 저장되었기 때문에, 비즈니스 요건이 바뀌어 새로운 관점의 분석이 필요해지면 원천 시스템부터 데이터를 다시 추출(Re-extract)해야 하는 비효율이 발생했습니다.

### 3.1 스토리지와 컴퓨팅의 분리: 클라우드 데이터 웨어하우스의 혁명

이러한 물리적 인프라의 한계를 근본적으로 무너뜨린 것은 클라우드 컴퓨팅 기술의 비약적인 발전입니다. Amazon Redshift, Snowflake, Google BigQuery, Databricks 등 클라우드 네이티브 데이터 웨어하우스·레이크하우스 서비스가 등장하면서 데이터 아키텍처의 패러다임은 극적으로 전환되었습니다.

이들 클라우드 플랫폼의 핵심 혁신은 데이터 저장(Storage)과 연산(Compute) 자원을 완벽하게 분리(Decoupling)해, 각각을 독립적이고 탄력적으로 확장할 수 있게 한 것입니다. 클라우드 환경에서 스토리지 비용은 온프레미스 시대와 비교할 수 없을 만큼 급감했고, 수백 대의 노드를 병렬로 가동해 복잡한 연산을 단숨에 수행하는 사실상 무한에 가까운 컴퓨팅 자원이 제공되었습니다.

### 3.2 ELT(Extract, Load, Transform) 아키텍처로의 전환

클라우드 인프라의 풍부한 자원은 데이터 파이프라인의 논리적 순서를 완전히 뒤집어 놓았습니다. 목적지 서버의 컴퓨팅 성능이 압도적으로 향상되고 데이터 보관 비용이 저렴해지자, 데이터를 굳이 별도의 외부 서버에서 변환한 뒤 적재할 필요가 사라진 것입니다. 원시 데이터를 다양한 원천 시스템에서 추출해 그 형태 그대로 유연한 클라우드 스토리지(데이터 레이크 또는 클라우드 웨어하우스)에 우선 적재(Load)한 뒤, 웨어하우스 내부의 강력한 분산 처리 엔진을 활용해 분석이 필요한 시점에 온디맨드(On-demand)로 변환(Transform)하는 ELT(Extract, Load, Transform) 방식이 새로운 산업 표준으로 자리 잡았습니다.

<!-- 다이어그램 제안: ETL과 ELT의 데이터 흐름 비교 — 변환(T) 단계가 목적지 외부냐 내부냐의 위치 차이 -->

| 비교 항목 | ETL (Extract, Transform, Load) | ELT (Extract, Load, Transform) |
|---|---|---|
| 작업 순서 | 추출 → 변환 → 적재 | 추출 → 적재 → 변환 |
| 변환의 주체·위치 | 전용 변환 서버 또는 외부 스테이징 환경 (목적지 외부) | 목적지 클라우드 웨어하우스·데이터 레이크 내부 |
| 처리 가능한 데이터 포맷 | 구조화된 정형 데이터에 최적화 | 반정형·비정형 등 유연한 포맷 수용 |
| 데이터 보존·재처리 | 변환 결과만 저장 → 원본 유실, 재처리 시 소스 재추출 필요 | 원시 데이터가 목적지에 영구 보존 → 소스 접근 없이 즉시 재처리 |
| 가장 적합한 인프라 | 확장이 제한적인 온프레미스 레거시 시스템 | 탄력적으로 확장되는 클라우드 기반 환경 |

ELT의 부상은 기업에 전례 없는 데이터 가용성과 유연성을 안겨 주었습니다. 원시 데이터가 가공되지 않은 상태로 보존되므로, 분석 모델이나 비즈니스 로직이 바뀌어도 소스 시스템에 다시 접근할 필요 없이 클라우드 환경 안에서 즉각적인 히스토리컬 데이터 재처리(Historical reprocessing)가 가능해졌습니다.

이러한 ELT 패러다임은 현대 데이터 레이크하우스 아키텍처의 핵심인 메달리온 아키텍처(Medallion Architecture, Bronze/Silver/Gold) 패턴을 정립하는 기반이 되었습니다.

- **Bronze(Raw Zone):** 소스 시스템의 데이터가 EL(Extract & Load) 과정을 통해 어떠한 수정도 없이 그대로 적재됩니다. 완벽한 감사 추적성(Auditability)과 무제한적인 재처리를 보장합니다.
- **Silver(Cleaned Zone):** 클라우드 웨어하우스 내부의 연산 능력을 활용해 변환(Transformation) 작업이 시작됩니다. 데이터 타입 캐스팅, 중복 제거, 형식 표준화 같은 정제 작업이 이루어집니다.
- **Gold(Business-ready Zone):** 대시보드 시각화, 비즈니스 리포팅, 머신러닝 피처(Feature) 모델링 등 최종 사용자의 목적에 맞게 고도로 집계되고 최적화된 데이터가 생성됩니다.

### 3.3 배치 처리에서 실시간 스트리밍(Streaming) ETL로

ELT로의 전환과 함께, 데이터 처리 주기에서도 배치(Batch) 방식에서 실시간 스트리밍(Real-time Streaming) 방식으로의 근본적인 진화가 일어났습니다. 전통적인 배치 파이프라인은 일정한 시간 간격(예: 자정, 매시간)으로 데이터를 수집·처리했지만, 초개인화된 실시간 추천, 금융 사기 탐지, 생산 라인의 즉각적인 이상 탐지 등을 요구하는 현대 비즈니스 환경에서는 허용될 수 없는 지연(Latency)을 낳았습니다.

이에 대응해 Apache Kafka 같은 오픈소스 스트림 처리 플랫폼이 도입되면서, 데이터가 생성되는 즉시 연속적인 흐름 속에서 변환을 수행하고 목적지로 보내는 스트리밍 ETL 아키텍처가 확산되었습니다. 다만 실시간 스트리밍은 시스템 설계자에게 극도의 기술적 난관을 요구합니다. 시스템은 24시간 무중단으로 데이터를 수집해야 하므로 고도의 내결함성(Fault-tolerance) 인프라가 필수적이고, 데이터가 눈 깜짝할 사이에 처리되어야 하므로 배치 방식처럼 사전에 정합성을 맞출 시간을 확보하기 어렵습니다. 또한 시스템 장애가 발생하면, 배치 시스템은 소스에서 데이터를 다시 가져와 작업을 재실행하면 되지만, 스트리밍 환경에서는 별도의 버퍼나 백업 저장소가 없는 한 전송 중이던 데이터가 영구히 소실될 위험이 있습니다. 그럼에도 즉각적인 데이터 기반 행동(Action)이 주는 경쟁력 때문에, 대규모 클라우드 자원을 활용한 실시간 파이프라인 구축 요구는 멈추지 않았습니다.

## 4. 분석 엔지니어링의 태동과 dbt의 부상

클라우드 컴퓨팅과 ELT 아키텍처의 부상은 물리적인 데이터 저장소와 연산 계층의 병목을 거의 완벽하게 해소해 주었습니다. 데이터를 클라우드 웨어하우스에 우선 밀어 넣는(Load) 일까지는 Fivetran, Airbyte 같은 자동화 파이프라인 도구로 손쉽게 해결되었습니다. 문제는 그다음이었습니다. 초기의 ELT에서 변환(T)은 거대한 저장 프로시저(Stored Procedure)나 여기저기 흩어진 수동 SQL 스크립트에 의존했고, 의존성 관리도 버전 관리도 테스트도 없는 '야생의 시대'에 가까웠습니다. 데이터 파이프라인의 마지막이자 가장 중요한 과정, 즉 비즈니스 로직에 맞춰 데이터를 다듬는 변환(Transform) 단계에서 조직적·인적 자원 측면의 심각한 마찰이 발생하기 시작한 것입니다.

### 4.1 데이터 엔지니어와 분석가 간의 구조적 마찰

2015년에서 2016년 무렵, 데이터 조직 내부에는 분명한 직군 간의 단절이 존재했습니다. 클라우드 웨어하우스 내부에서 데이터를 정제하고 복잡한 의존성을 관리하며 파이프라인(DAG) 코드를 작성하는 일은, 분산 시스템과 Java·Scala·Python 같은 고급 프로그래밍 언어에 능통한 '데이터 엔지니어'들의 전유물이었습니다.

반면, 회사의 마케팅 캠페인 로직, 매출 인식 기준, 고객 이탈률 산정 방식 등 핵심 비즈니스 맥락을 가장 정확히 이해하는 '데이터 분석가'들은 SQL 쿼리에는 매우 능통했지만, 파이프라인을 구축할 권한도 적절한 도구도 없었습니다. 분석가는 새로운 지표를 보고서에 추가하려면 데이터 엔지니어에게 요구사항을 정의한 티켓(Ticket)을 발행하고, 엔지니어가 변환 로직을 코드로 구현해 배포해 줄 때까지 기약 없이 차례를 기다려야 했습니다. 이른바 '티켓팅 지옥(Ticketing Hell)'입니다. 당시의 상용 변환 도구들은 여전히 무겁고 폐쇄적이었으며 분석가가 접근하기에는 학습 곡선이 너무 가팔랐습니다. 클라우드가 제공하는 무한한 민첩성을 정작 조직 구조가 가로막는, 역설적인 상황이 펼쳐진 것입니다.

### 4.2 Fishtown Analytics의 창업과 dbt의 탄생

이러한 업계의 구조적 모순을 정면으로 돌파한 도구가 바로 dbt(data build tool)입니다. dbt의 탄생 이야기는 2016년 펜실베이니아주 필라델피아로 거슬러 올라갑니다. 벤처 투자를 받은 스타트업 생태계에서 7년간 주 70시간 넘는 격무에 시달리며 번아웃(Burnout)을 겪었던 트리스탄 핸디(Tristan Handy)는, 동료였던 드류 배닌(Drew Banin), 코너 맥아더(Connor McArthur)와 함께 데이터 분석·비즈니스 인텔리전스 기업 RJMetrics에서의 경험을 바탕으로 새로운 방향을 모색했습니다.

세 창립자는 RJMetrics에서 일하며 데이터 엔지니어링 인프라와 비즈니스 분석 사이의 끊임없는 마찰을 목격했고, 분석가가 소프트웨어 엔지니어처럼 파이프라인을 다룰 수 있어야 한다는 강력한 비전을 품게 되었습니다. 2016년, 이들은 벤처 자금 없이 'Fishtown Analytics'라는 3인 컨설팅 회사를 설립했습니다. 외부 투자를 받아 무리하게 성장하는 대신, 컨설팅 서비스로 번 수익으로 회사를 부트스트래핑(Bootstrapping)하며 초기 아이디어를 실제 고객의 파이프라인에 적용해 보는 길을 택한 것입니다. 이때 고객사의 클라우드 기반 모던 데이터 스택을 구축해 주면서 내부적으로 활용하려고 고안한 파이프라인 모듈화 도구가 바로 초기 버전의 dbt였습니다.

### 4.3 dbt의 혁신: 분석 엔지니어링의 창시

dbt가 지향한 목표는 명확했습니다. ELT 파이프라인에서 'E(추출)'와 'L(적재)'은 완전히 배제하고, 클라우드 웨어하우스 내부에서 이루어지는 'T(변환)'에만 철저히 집중하는 것이었습니다. dbt는 데이터 분석가가 소프트웨어 엔지니어링의 모범 사례(버전 관리, 모듈화, 테스트, CI/CD)를 적용해 데이터 변환을 주도할 수 있도록 지원했고, 그 결과 '분석 엔지니어(Analytics Engineer)'라는 완전히 새로운 직군이 시장에 등장했습니다.

dbt가 이러한 요구를 충족하기 위해 채택한 기술적 형태와 핵심 원칙은 다음과 같습니다.

- **SQL 중심의 개발 — 접근성의 극대화:** 데이터 엔지니어가 아닌 일반 비즈니스 분석가도 쉽게 쓸 수 있도록, dbt는 변환 모델의 작성 언어로 데이터계의 공용어인 SQL을 채택했습니다. 분석가는 복잡한 데이터 조작 언어(DML)나 데이터 정의 언어(DDL)를 고민할 필요 없이 핵심 변환 로직인 `SELECT` 쿼리만 작성하면 되고, 테이블이나 뷰(View)를 생성하는 작업은 dbt가 백그라운드에서 대신 처리합니다.
- **Jinja 템플릿과 매크로를 통한 모듈화:** 순수 SQL이 가진 정적이고 반복적인 한계를 극복하기 위해, dbt는 파이썬 기반 템플릿 엔진 Jinja를 SQL 구문 안에 통합했습니다. 이를 통해 반복문(`for`)을 순회하거나 조건문(`if`)을 삽입할 수 있고, 반복되는 SQL 로직을 매크로(Macro) 함수로 추상화해 코드 재사용성을 소프트웨어 개발 수준으로 끌어올렸습니다.
- **자동화된 DAG 기반 의존성 관리:** 분석가가 `ref('모델명')` 함수로 다른 변환 테이블을 참조하기만 하면, dbt는 프로젝트 내 모든 모델 간의 관계를 분석해 '방향성 비순환 그래프(DAG, Directed Acyclic Graph)'를 자동으로 생성합니다. 데이터 실행 순서를 완벽하게 보장해 개발자의 수동 스케줄링 부담을 없앤 것입니다.
- **품질 검증과 문서화 통합:** 각 데이터 모델에 대해 기본 키 식별, Null 값 부재 검증, 참조 무결성 확인 같은 테스트를 YAML 설정 파일에서 손쉽게 정의할 수 있게 했고, 데이터 리니지(Lineage) 시각화 문서를 자동 생성해 데이터의 신뢰성을 확보했습니다.

초기 dbt는 Casper, Kickstarter 등 뉴욕의 진취적인 기술 스타트업을 중심으로 빠르게 입소문을 탔습니다. GitHub 저장소와 Slack 커뮤니티를 중심으로 모인 실무자들의 폭발적인 자발적 확산에 힘입어 dbt 사용 조직은 매월 10%씩 기하급수적으로 성장했습니다.

이러한 강력한 오픈소스 생태계를 기반으로 포춘 500대 기업들이 엔터프라이즈 지원을 요청하기 시작했고, Fishtown Analytics는 컨설팅 서비스 중심에서 클라우드 기반 관리형 서비스형 소프트웨어(SaaS)인 dbt Cloud 중심의 제품 주도 성장(Product-led growth) 모델로 완전히 탈바꿈했습니다. 2020년 1,290만 달러 규모의 시리즈 A 투자로 성장에 가속도를 붙인 회사는 2021년 사명을 dbt Labs로 변경했고, 2022년에는 2억 2,200만 달러 규모의 시리즈 D 투자를 유치하며 기업 가치 42억 달러에 이르는, 모던 데이터 스택의 사실상 표준(De-facto standard)으로 자리매김했습니다.

## 5. 차세대 데이터 변환 아키텍처: dbt의 한계와 SQLMesh, SDF의 등장

dbt가 클라우드 네이티브 변환의 업계 표준으로 등극하며 수만 개 조직에 도입되었지만, 엔터프라이즈 환경에서 데이터 모델이 수천 개로 늘고 의존성이 극도로 복잡해지면서 dbt Core 아키텍처의 태생적인 한계들이 드러나기 시작했습니다. 분석 엔지니어들은 대규모 환경에서 파이프라인을 안정적이고 경제적으로 운영하기 위해 더 진보된 통제력과 최적화를 요구했고, 이에 부응해 SQLMesh, SDF(Semantic Data Fabric) 같은 차세대 프레임워크가 강력한 대안으로 부상했습니다.

### 5.1 dbt 아키텍처의 구조적 한계

dbt는 본질적으로 빠른 개발과 유연성에 초점을 맞춘 도구입니다. 그래서 대규모 데이터 파이프라인의 복잡한 연산 최적화와 상태 추적 측면에서는 약점을 드러냈습니다.

- **상태 비저장(Stateless) 기반의 설계:** 오픈소스 dbt Core는 기본적으로 실행 이력이나 과거 상태를 내장 저장소에 추적하지 않는 '상태 비저장' 아키텍처입니다. 테이블을 전체 새로고침(Full-refresh)하지 않고 새로 추가된 레코드만 업데이트하는 증분 모델(Incremental Model)을 구성하려면, 분석가가 `is_incremental()` 매크로로 상태 판별 로직을 SQL 안에 직접 작성해야 합니다. 또한 코드 변경 시 영향을 받는 모델만 부분 빌드(Partial build)하려면 과거 실행에서 생성된 매니페스트(Manifest) 파일에 수동으로 의존해야 하는 번거로움이 따릅니다.
- **단순 문자열(String) 기반의 SQL 처리:** dbt는 Jinja 템플릿을 해석한 결과를 단순한 텍스트 문자열로 취급해 데이터베이스 엔진에 밀어 넣습니다. 이는 dbt 자체가 SQL 구문을 의미론적(Semantically)으로 이해하지 못한다는 뜻입니다. 쉼표 누락, 컬럼 이름 오타, 타입 불일치 같은 명백한 문법 오류조차 dbt 컴파일 단계에서는 잡히지 않고, 연산 비용을 지불해 가며 런타임에 쿼리를 실행하다 실패하고 나서야 비로소 오류를 인지하게 됩니다. 컬럼 단위의 정밀한 데이터 리니지(Column-level Lineage) 역시 문자열 파싱의 한계 탓에, 외부 유료 도구를 쓰지 않는 한 자체 추론이 불가능합니다.
- **비효율적인 개발 환경 복제 비용:** dbt에서 프로덕션과 격리된 개발·테스트 환경을 구축하는 전통적인 방법은, 전체 웨어하우스 데이터나 모델을 새로운 스키마에 런타임에 물리적으로 복사하거나 전부 다시 연산하는 것입니다. 데이터 규모가 커질수록 막대한 클라우드 컴퓨팅·스토리지 비용을 낭비하게 되고, 개발 주기도 심각하게 지연됩니다.

특히 컬럼 단위 리니지의 부재는 다음과 같은 구체적인 실무 문제로 이어집니다.

1. **사전 영향도 파악 불가:** 특정 컬럼을 삭제했을 때 어느 대시보드가 깨지는지 미리 알 수 없습니다.
2. **디버깅의 어려움:** 지표 수치가 틀어졌을 때, 원인을 찾으려면 수많은 SQL 모델을 손으로 역추적해야 합니다.
3. **보안 사각지대:** 개인정보(전화번호 등)가 담긴 컬럼이 어느 테이블로 흘러갔는지 추적하기 어렵습니다.
4. **비용 낭비:** 더 이상 쓰이지 않는 컬럼을 안심하고 제거하지 못해, 불필요한 연산·저장 비용이 계속 발생합니다.

### 5.2 대안 1: SQLMesh와 구문 트리(AST) 기반의 혁신

Tobiko Data가 주도하는 오픈소스 프로젝트 SQLMesh는 dbt의 이러한 약점들을 정조준해, 처음부터 견고한 데이터옵스(DataOps) 프레임워크로 설계되었습니다.

- **SQLGlot을 활용한 시맨틱 파싱(Semantic Parsing):** SQLMesh는 파이썬 기반의 강력한 SQL 파서 SQLGlot을 내장해, SQL 쿼리를 단순 텍스트가 아닌 '추상 구문 트리(AST, Abstract Syntax Tree)' 단위로 분해하고 구조적으로 완전히 이해합니다. 덕분에 코드를 작성하는 즉시 컴파일 타임에 문법 오류를 잡아내고, 별도 설정 없이도 완벽한 컬럼 단위 리니지를 자동으로 추론합니다.
- **상태 저장(Stateful) 기반의 지능적 의존성 관리:** SQLMesh는 실행 이력과 프로젝트 메타데이터를 웨어하우스나 별도의 상태 DB에 자동으로 기록합니다. 사용자가 코드를 수정하면 AST 수준에서 무엇이 바뀌었는지 정확히 파악해, 하위 의존성을 추적하고 변경의 영향을 받는 모델만 골라 연산하는 실행 계획(Plan)을 제시합니다.
- **가상 데이터 환경(VDE)과 포인터 교체 프로모션:** SQLMesh의 가장 강력한 무기 중 하나는 '가상 데이터 환경(Virtual Data Environments)'입니다. 개발 환경을 만들 때 데이터를 물리적으로 복제하는 대신, 뷰(View)와 테이블의 상태 식별자 해시값을 이용한 논리적 포인터 참조를 활용합니다. 덕분에 수 테라바이트 규모의 데이터 환경을 복제하는 데 추가 비용도 시간도 들지 않습니다. 개발한 로직을 프로덕션으로 배포(Promotion)할 때도 데이터 재연산 없이 포인터 위치만 바꾸므로, 전환과 롤백(Rollback)이 즉각적이고 안전합니다.

### 5.3 대안 2: SDF(Semantic Data Fabric)와 DuckDB 결합의 진화

SQLMesh와 비슷한 철학을 공유하며 주목받은 또 다른 플랫폼은 SDF(Semantic Data Fabric)입니다. SDF는 데이터를 단순 번역하는 수준을 넘어, 논리적 실행 계획(Logical Plan Layer) 단계에서 완전히 컴파일하는 방식을 취합니다. 이를 통해 데이터 타입 바인딩과 스키마 수준의 정교한 유효성 검증을 네이티브하게 제공해 파이프라인의 안전성을 극대화합니다.

또한 클라우드 자원을 낭비하지 않고 데이터 파이프라인을 테스트하려는 요구는, DuckDB와 결합된 로컬 우선(Local-first) 개발 트렌드를 낳았습니다. SQLite와 비슷하게 작동하지만 분석용(OLAP)으로 설계된 인메모리 내장형 데이터베이스 DuckDB는, 파케이(Parquet)나 CSV 파일과 직접 상호작용하며 매우 빠른 연산 속도를 보여 줍니다. 분석가들은 `dbt-duckdb` 어댑터나 SQLMesh를 DuckDB와 연결해, 값비싼 Snowflake나 BigQuery 가상 웨어하우스를 가동하지 않고도 자신의 노트북에서 단 몇 초 만에 데이터 모델링 변환과 유닛 테스트를 수행하는 'Modern Data Stack In A Box'를 실현하고 있습니다.

| 비교 항목 | dbt | SQLMesh · SDF |
|---|---|---|
| 코드 해석 방식 | Jinja 처리 후 단순 텍스트(String)로 엔진에 전달 | SQL 파서(SQLGlot 등) 기반 AST 시맨틱 파싱·컴파일 |
| 상태 관리 | 기본적으로 상태 비저장(Stateless), 매니페스트·수동 매크로 의존 | 상태 저장(Stateful) 내장, 실행 이력 기반 점진적 연산 계획 |
| 개발 환경 구성 | 데이터를 새 스키마에 물리적으로 복제·재연산 | 포인터 기반 가상 데이터 환경(VDE), 무비용 논리적 격리 |
| 리니지 추론 | 테이블 단위까지. 컬럼 단위는 오픈소스 기본 미지원 | AST 분석으로 컬럼 단위 리니지 자동 생성 |

흥미롭게도 이 차세대 도구들의 행보는 '독립된 경쟁자'로 끝나지 않았습니다. 2025년 들어 dbt Labs는 SDF Labs를 인수해 그 SQL 이해(comprehension) 기술을 자사 제품에 흡수했고, SQLMesh를 이끌던 Tobiko Data 역시 데이터 통합 기업 Fivetran에 인수되었습니다. AST 기반 시맨틱 파싱이라는 혁신은 별도의 진영으로 남기보다, 기존 모던 데이터 스택 안으로 빠르게 빨려 들어가고 있는 셈입니다.

## 6. 시맨틱 레이어와 온톨로지의 융합

ELT 인프라의 확충, 그리고 dbt와 SQLMesh를 필두로 한 변환 로직 자동화가 일정 궤도에 오르면서, 데이터 엔지니어링 업계의 주요 관심사가 옮겨 가기 시작했습니다. "물리적인 데이터를 어떻게 효율적으로 이동시키고 가공할 것인가"라는 문제를 어느 정도 극복한 뒤, "이 데이터가 담은 의미(Meaning)를 전사적으로 어떻게 통일하고, 나아가 AI가 이를 어떻게 이해하게 만들 것인가"라는 문제와 마주하게 된 것입니다. 이 지점에서 현대 데이터 아키텍처의 정점인 시맨틱 레이어(Semantic Layer)와 온톨로지(Ontology)의 융합 논의가 본격적으로 시작됩니다.

### 6.1 시맨틱 레이어의 본질: '측정' 메커니즘의 통제

데이터 파이프라인이 정비되어 완벽히 깨끗한 데이터가 웨어하우스에 준비되었어도, 비즈니스 부서의 혼란은 여전했습니다. 영업 부서가 Tableau에서 보는 '월간 활성 고객 매출'과, 재무 부서가 Power BI에서 조회하는 '매출' 지표가 서로 다른 수치를 가리키는 일이 빈번하게 벌어졌기 때문입니다. 데이터베이스의 물리적 스키마(영문 테이블명, 조인 키, 결측치 처리 로직 등)와 비즈니스 사용자가 직관적으로 쓰는 용어(수익, 이탈률 등) 사이의 추상화 간극을 통제할 중앙 거버넌스가 없었던 탓입니다.

시맨틱 레이어는 물리적 데이터베이스와 모든 다운스트림 소비자(BI 도구, 웹 애플리케이션, REST API 등) 사이에 위치하는 얇은 추상화 계층입니다. 그 존재 목적은 '매출' 같은 비즈니스 지표(Metrics)가 조직 내 어디에서 쿼리되든 항상 100% 동일한 필터 조건과 산식으로 계산되도록 통제하는 것입니다.

dbt Labs는 이러한 요구에 부응하기 위해 2023년 시맨틱 엔진 스타트업 Transform을 인수했고, MetricFlow 프레임워크를 기반으로 한 dbt Semantic Layer를 출시했습니다. 이를 통해 분석 엔지니어들은 기존의 테이블 변환(dbt Core)뿐 아니라 지표의 산출 공식까지도 YAML 형태의 코드(Metrics as Code)로 정의하고 Git으로 버전 관리할 수 있게 되었습니다. 그 실용적 효과 중 하나는 분석의 유연성입니다. 집계 테이블을 미리 일일이 만들어 두지 않아도, 분석가가 새로운 분석 차원(Dimension)을 던지면 시맨틱 레이어가 통제된 산식을 바탕으로 SQL을 동적으로 생성해 줍니다.

요약하면, 시맨틱 레이어의 핵심 역할은 "지표를 어떻게 측정(How to measure)할 것인가"에 대한 일관된 계산 규칙과 산식을 결정론적(Deterministic)으로 제공하는 것입니다.

### 6.2 온톨로지의 본질: '의미'의 부여와 '추론' 능력

반면 온톨로지(Ontology)는 시맨틱 레이어와 이름의 뉘앙스만 비슷할 뿐, 기원과 근본 목적이 완전히 다릅니다. 온톨로지는 1990년대 인공지능 연구와 지식 관리, 문헌정보학 영역에서 발전한 개념으로, 1993년 Tom Gruber가 "개념화(Conceptualization)에 대한 명시적 명세(explicit specification of a conceptualization)"로 정의했습니다. 이후 Studer 등(1998)이 여기에 '공식적이고(formal)' '공유된(shared)'이라는 조건을 더해, 오늘날 흔히 인용되는 "공유된 개념화에 대한 공식적이고 명시적인 명세"라는 형태로 다듬어졌습니다.

데이터 모델링 관점에서 온톨로지는, 특정 비즈니스 도메인 안의 개념(Concepts)과 그들 사이의 복잡한 관계(Relationships), 그리고 이를 얽어매는 논리적 규칙(Axioms/Rules)을 기계가 명확히 읽고 해석할 수 있는(Machine-readable) 형식으로 정의한 거대한 청사진입니다. 이러한 온톨로지는 Resource Description Framework(RDF)나 Web Ontology Language(OWL) 같은 W3C 시맨틱 웹 표준 규격으로 작성됩니다.

온톨로지는 크게 세 가지 요소로 도메인을 묘사합니다.

- **클래스(Classes) — 명사:** 도메인 안에 존재하는 사물이나 개념의 타입 (예: Customer, Product, Order).
- **관계·속성(Properties) — 동사:** 클래스들이 어떻게 연결되는지를 정의 (예: Customer는 Order를 "생성한다", Order는 Product를 "포함한다").
- **공리·제약(Axioms/Rules):** 도메인을 지배하는 논리적 절대 법칙 (예: 하나의 Order 객체는 반드시 최소 1개 이상의 Product를 가져야 한다).

그리고 지식 그래프(Knowledge Graph)는, 온톨로지라는 설계도(Schema)를 바탕으로 실제 데이터 소스들이 노드와 엣지로 연결되어 구현된 물리적 데이터 집합체를 의미합니다.

시맨틱 레이어가 "수익을 더하고 나누는 수학적 방식"에 집중한다면, 온톨로지는 "수익이라는 개념 자체가 기업의 고객 세그먼트, 규제 요건, 산업 분류 코드와 어떤 본질적 연관성을 지니는가"에 집중합니다. 두 계층의 가장 결정적인 기술적 차이는 바로 '자동화된 연역적 추론(Logical Inference)' 능력에 있습니다. 예를 들어 온톨로지 안에 "플래티넘 고객은 연간 기여 수익이 100만 달러 이상인 자이다"라는 규칙(Rule)이 있고, 웨어하우스 데이터에 "Acme Corp의 누적 수익이 200만 달러다"라는 사실(Fact)이 입력되면, 기계는 인간이 명시적으로 작성한 `if-else` 코드 없이도 자체 논리 연산을 통해 Acme Corp를 '플래티넘 고객'으로 자동 분류하고 새로운 사실을 도출해 냅니다. 시맨틱 레이어나 기존 BI 도구는 사전에 정의된 SQL 산출식을 수동적으로 실행할 뿐, 스스로 지식을 창출하거나 관계를 연역하지는 못합니다.

| 특성·기능 | 시맨틱 레이어 (Semantic Layer) | 온톨로지 (Ontology) |
|---|---|---|
| 도입 목적 | 비즈니스 지표 산출의 일관성 — "어떻게 측정할 것인가" | 도메인 지식의 모델링과 관계 규명 — "무엇을 의미하는가" |
| 주요 산출물 | 통제된 결정론적 SQL 쿼리, 일관된 KPI 수치 | 지식 그래프, 개념 모델, 추론 결과 |
| 자동 추론 능력 | 없음 — 사전에 정의된 계산만 실행 | 형식 논리 규칙으로 새로운 사실을 자체 연역 |
| 관련 기술·표준 | YAML, SQL, LookML, DAX | RDF, OWL, SPARQL (W3C 시맨틱 웹 표준) |
| 구축 난이도·기간 | 2~6개월 내 배포 가능, 빠른 ROI | 6~18개월 소요, 고도의 논리 설계와 전문 인력(Ontologist) 필요 |

### 6.3 dbt와 온톨로지: 대체재인가, 필수적 결합인가?

dbt(시맨틱 레이어)와 온톨로지는 서로 무관하거나 서로를 배제하는 대체재 관계가 아닙니다. 오히려 고도로 진보된 최신 엔터프라이즈 AI 시스템에서는 필수 불가결한 상호보완적(Complementary) 관계로 엮입니다. dbt는 온톨로지를 대체할 수 없고, 온톨로지 지식 그래프 단독으로는 빠르고 일관된 비즈니스 지표의 수치 집계 성능을 보장할 수 없습니다.

과거 인간 분석가가 데이터를 소비하던 시절에는, 대시보드의 용어가 모호해도 동료에게 메신저로 물어보거나 회의를 거쳐 스스로 문맥을 보정할 수 있었습니다. 그러나 AI 에이전트는 불분명한 데이터 테이블 구조나 지표 이면에 깔린 배경지식을 스스로 상상하지 못합니다. LLM에 기업 데이터에 대한 직접적인 쿼리 작성 권한(Text-to-SQL)을 부여하면, 모델은 컬럼명이나 조인 경로를 제멋대로 추측해 그럴싸하지만 완전히 틀린 수치를 내놓는 치명적인 '환각 현상(Hallucination)'을 빈번하게 일으킵니다.

이러한 결함을 극복하기 위해, 최신 데이터 스택은 온톨로지와 시맨틱 레이어를 하나의 파이프라인 안에서 융합하는 방식을 채택했습니다.

1. 사용자가 AI에게 "아시아 지역 주요 고객들의 전년 대비 수익 변화는?"이라고 질문을 던집니다.
2. AI 에이전트는 먼저 온톨로지(지식 그래프) 계층을 탐색해, '주요 고객'이 어떤 기업 규모와 계약 조건을 만족하는 엔티티(Entity)인지, '아시아 지역'이 어떤 국가 코드들과 관계망을 이루는지에 대한 풍부한 구조적 문맥(Context)과 의미를 추론합니다.
3. 맥락을 완전히 파악한 AI는 숫자를 직접 계산하려 들지 않고, 시맨틱 레이어(dbt)에 사전 정의된 결정론적 지표 호출 API(Metrics Query API)를 트리거합니다. 시맨틱 레이어는 정확히 조인되고 통제된 SQL을 실행해 오차 없는 수익 수치를 반환합니다.

즉, 시맨틱 레이어가 "계산과 정밀성(Calculation & Precision)"을 책임지고, 온톨로지가 "이해와 추론(Comprehension & Reasoning)"을 책임짐으로써 완벽한 협업이 이루어집니다.

## 7. 생성형 AI 시대의 SOTA 아키텍처: 컨텍스트 레이어와 MCP

시맨틱 레이어와 온톨로지의 개념적 결합을 실제 기술 프레임워크로 구현해 기업의 AI 에이전트에게 공급하는 과정에서, 인프라 수준의 거대한 도약이 일어나고 있습니다. 이 도약의 두 축은 컨텍스트 레이어(Context Layer) 메타데이터 아키텍처의 구축과, Anthropic 등이 주도해 제정한 개방형 연동 표준 Model Context Protocol(MCP)의 활용입니다.

### 7.1 Atlan과 엔터프라이즈 컨텍스트 레이어의 부상

기존의 메타데이터 플랫폼이 단순한 데이터 카탈로그 역할에 머물렀다면, Atlan 같은 최신 데이터 지능 플랫폼은 기업 전반에 분산된 시스템 지식을 하나의 살아 있는 '엔터프라이즈 데이터 그래프(Enterprise Data Graph)'로 통합하는 컨텍스트 레이어를 제안합니다.

단순히 지표의 계산식을 정의하는 시맨틱 레이어를 넘어, 컨텍스트 레이어는 데이터의 혈통(Lineage), 기업 내부의 표준 운영 절차(SOP), 데이터 자산의 품질 등급, 접근 권한 거버넌스, 그리고 특정 데이터를 과거 경영진이 어떻게 의사결정에 활용했는지에 대한 이력(Decision Precedents)까지 모두 포함하는 상위 집합(Superset)의 역할을 합니다. Atlan은 이러한 풍부한 컨텍스트를 수작업으로 구축하는 대신, 4단계 파이프라인을 통해 자동화한다고 설명합니다.

- **Unify(통합):** Snowflake, dbt, Airflow 등 80개 이상의 소스에서 SQL 쿼리 기록과 파이프라인 메타데이터를 수집해 데이터 그래프를 구축합니다.
- **Bootstrap(부트스트랩):** AI 컨텍스트 에이전트가 쿼리 이력을 읽어 초기 자산 설명, 지표, 온톨로지 초안을 상당 부분 자동 생성합니다.
- **Collaborate(협업):** 도메인 전문가(인간)가 엣지 케이스를 수정하고 지식의 정확성을 최종 인증(Certify)합니다.
- **Activate(활성화):** 검증된 컨텍스트가 다운스트림 AI 애플리케이션에 즉시 노출됩니다.

이렇게 맥락이 풍부한 지식 그래프를 RAG(Retrieval-Augmented Generation) 시스템과 결합했을 때의 효과는 여러 사례에서 보고되고 있습니다. 예를 들어 공중보건 도메인의 RAG 연구인 MEGA-RAG(《Frontiers in Public Health》, 2025)는 의료 지식 그래프(CPubMed-KG)로 검색을 증강했을 때 환각률을 40% 이상 줄였다고 보고했습니다. 다만 이는 특정 도메인에 한정된 결과라는 점은 감안할 필요가 있습니다. 또한 데이터 카탈로그 벤더 Atlan의 자체 벤치마크에서는, AI 에이전트가 풍부한 컨텍스트 메타데이터에 접근했을 때 Text-to-SQL 정확도가 상대적으로 38% 향상되었다는 결과(절대 수치로는 16.1%에서 22.2%)를 제시한 바 있습니다. 두 수치 모두 엄밀한 일반화에는 한계가 있지만, 컨텍스트의 양과 질이 AI의 데이터 해석 정확도에 실질적인 영향을 준다는 방향성만큼은 일관되게 가리키고 있습니다.

### 7.2 Model Context Protocol(MCP)과 dbt MCP Server의 결합

컨텍스트와 시맨틱 데이터를 잘 구축해 두었더라도, 시중에 쏟아지는 수많은 언어 모델과 AI 비서(ChatGPT, Claude, Copilot 등)가 이 데이터에 제각각 접근하기 위해 매번 1:1 커스텀 커넥터를 개발하는 것은 유지보수 측면에서 악몽과 같습니다. 이러한 극심한 통합 복잡성을 해소하기 위해 2024년 말 등장한 개방형 표준 규격이 바로 Model Context Protocol(MCP)입니다.

MCP는 AI 에이전트가 외부의 데이터베이스, 애플리케이션 로직, 시맨틱 레이어에 일관되고 안전하게 접근할 수 있게 해 주는 'AI를 위한 범용 USB-C 포트'에 비유됩니다. MCP 환경에서 AI는 데이터베이스를 무단으로 휘젓고 다니며 임의의 원시 SQL을 작성할 자유를 갖지 못합니다. 대신, 기업이 사전에 노출해 둔 엄격히 정의된 함수 형태의 '도구(Tools)' 메뉴만 선택적으로 호출해 결과를 전달받는 안전한 방식으로 작동합니다.

이러한 흐름에 발맞춰 dbt Labs는 AI 에이전트와 기업 데이터 사이의 다리 역할을 하는 dbt MCP Server를 공식 출시했습니다. dbt MCP Server는 dbt 프로젝트 내부에 구축된 방대한 지식 — 변환 모델의 정의, 테스트 내역, 의존성 리니지, 메타데이터 문서, 그리고 가장 핵심적인 시맨틱 레이어 지표 정의 — 을 MCP 표준 도구 형태로 래핑(Wrapping)해 AI 에이전트에게 제공합니다.

- **기능 노출의 세분화:** dbt MCP Server는 프로젝트 내 모든 모델 목록을 파악하는 '모델 탐색(Model Exploration)' 도구, 상류와 하류의 영향을 분석하는 '리니지 내비게이션(Lineage Navigation)' 도구, 그리고 가장 중요한 '시맨틱 레이어 쿼리(Semantic Layer Query)' 도구를 제공합니다.
- **배포 형태의 이원화:** 개발자 환경을 위해 CLI 명령어를 직접 실행하는 '로컬(Local) MCP 서버'와, 별도 설치 없이 HTTP 연결을 통해 비즈니스 애플리케이션의 데이터 소비·질의 응답에 특화된 '원격(Remote) MCP 서버'를 함께 제공해, 보안과 접근성을 동시에 충족합니다.

결과적으로 MCP라는 표준화된 터널을 통해, 온톨로지와 시맨틱 레이어라는 강력한 컨텍스트 인프라가 AI 에이전트의 두뇌(LLM)에 직접 연결됩니다. LLM은 더 이상 눈을 가린 상태가 아니며, 기업 데이터의 명확한 구조와 검증된 지표 계산기를 손에 쥐게 됩니다. 이는 경영진이 요구하는 신뢰할 수 있는 에이전틱 AI(Agentic AI) 시스템 구축을 향한 마지막 퍼즐 조각을 맞추는 도약입니다.

## 마치며

엔터프라이즈 데이터 파이프라인의 역사는 결국 "이질적인 시스템에 파편화된 데이터에서 어떻게 일관되고 통합된 비즈니스 가치를 끌어낼 것인가"라는 한 가지 물음에 대한 산업계의 끈질긴 답변이었습니다. 1960년대 자기 테이프의 순차 처리에서 출발한 데이터 관리 기술은, 1990년대 빌 인몬의 Prism Solutions를 필두로 한 상용 ETL 도구의 등장을 통해 하향식 통합의 틀을 갖췄습니다. 그러나 물리적 서버에 종속된 1세대 ETL의 폐쇄성과 고비용 구조는, 2010년대 클라우드 컴퓨팅 혁명과 스토리지·연산 분리 기술을 만나 무한히 확장 가능한 ELT 아키텍처로 그 패러다임을 완전히 뒤집었습니다.

흥미로운 점은, 클라우드가 컴퓨팅의 자유를 안겨 주자 이번에는 데이터를 변환하는 '사람들' 사이의 마찰이 새로운 병목으로 떠올랐다는 것입니다. 이 조직적 한계를 타파하기 위해 2016년 등장한 dbt는 SQL의 직관성, Jinja의 모듈화, 그리고 소프트웨어 엔지니어링의 규율(버전 관리·테스트·CI/CD)을 결합해 분석가에게 파이프라인을 직접 다룰 힘을 쥐여 주었고, '분석 엔지니어링'이라는 직군의 탄생을 이끌었습니다.

이 여정을 한 장면으로 압축하면 이렇게 말할 수 있습니다. 클라우드 데이터 웨어하우스가 무거운 데이터를 실어 나를 **트럭(ELT)**을 마련했다면, dbt Core는 분석가에게 그 트럭의 **운전대(SELECT 중심의 쉬운 변환)**를 쥐여 주며 해방시켰고, dbt Semantic Layer는 모두가 같은 목적지를 향하도록 **중앙 통제 내비게이션(표준화된 지표)**을 달아 준 셈입니다. 이후 시스템 복잡도가 한계에 다다르자 SQLMesh와 SDF 같은 AST 기반 차세대 도구들이 가상 데이터 환경과 로컬 우선 생태계로 이 트럭의 성능을 한 단계 더 끌어올렸습니다.

그리고 파이프라인 인프라가 완숙기에 접어든 지금, 데이터 생태계의 무게중심은 분명히 다음 단계로 옮겨 가고 있습니다. 이제 핵심 과제는 데이터를 정제해 저장하는 것을 넘어, 데이터의 '계산적 의미(시맨틱 레이어)'와 '구조적 개념·인과관계(온톨로지)'를 전사적으로 융합하는 것입니다. dbt MetricFlow가 보장하는 결정론적 측정의 신뢰성과, 컨텍스트 레이어가 제공하는 기계 추론 능력은 Model Context Protocol이라는 표준화된 다리를 건너 생성형 AI 에이전트의 두뇌로 전달됩니다. 데이터 엔지니어링의 역사는 데이터를 물리적으로 저장하고 옮기던 '배관공의 시대'를 지나, 전사의 지식과 논리를 엮어 내는 '인지적 인프라(Cognitive Infrastructure)의 시대'로 그다음 장을 넘기고 있습니다.

AI가 문서의 구조와 맥락을 어떻게 읽어 들이는지에 관심이 있다면, 같은 문제의식을 콘텐츠 작성 관점에서 다룬 [AI 친화적 콘텐츠 작성 가이드]({% post_url 2026-04-03-ai-friendly-content-writing-guide %})도 함께 참고할 만합니다.
